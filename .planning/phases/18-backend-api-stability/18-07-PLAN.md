**Issue — No request ID correlation:**
- When a request hits multiple services (e.g., webhook → plaidService → plaidDataService), there's no way to correlate log entries
- Production debugging requires matching timestamps, which is unreliable with concurrent requests
- Impact: Hard to trace request flows across services
- Fix: Add request ID middleware + propagate through logger

**Established patterns:**
- Winston structured logging with child loggers via createLogger(serviceName)
- Express middleware chain: helmet → cors → rate limit → auth → validate → route
- Metadata objects in log calls: `logger.info('message', { key: value })`

**Constraining decisions:**
- Phase 8 established Winston child logger pattern — must extend, not replace
- All services already use `createLogger('service-name')` — requestId needs to be additive

<task type="auto">
  <name>Task 1: Create request ID middleware</name>
  <files>backend-express/middleware/requestId.js, backend-express/server.js</files>
  <action>Create a new middleware file `backend-express/middleware/requestId.js`:

```javascript
const crypto = require('crypto');

function requestId(req, res, next) {
  // Use client-provided X-Request-ID if present (for distributed tracing), otherwise generate
  const id = req.headers['x-request-id'] || crypto.randomUUID();
  req.requestId = id;
  res.setHeader('X-Request-ID', id);
  next();
}

module.exports = requestId;
```

In server.js, add this middleware EARLY in the chain (after helmet, before cors/auth):
```javascript
const requestId = require('./middleware/requestId');
app.use(requestId);
```

Use `crypto.randomUUID()` (built into Node 19+, available in Node 20 which is the project target). Do NOT add uuid as a dependency — crypto.randomUUID() is sufficient.

Do NOT validate the format of client-provided X-Request-ID — just accept it for tracing continuity.</action>

  <done>Every request gets a unique ID. Client-provided IDs are passed through. Response includes X-Request-ID header for client correlation.</done>

```javascript
function createRequestLogger(baseLogger, requestId) {
  return baseLogger.child({ requestId });
}
```
Export it alongside `createLogger`.

2. In `middleware/auth.js` (or in the requestId middleware itself), attach a request logger to the request object:
```javascript
// In requestId middleware, after setting req.requestId:
const { createLogger, createRequestLogger } = require('../utils/logger');
const baseLogger = createLogger('request');
req.logger = createRequestLogger(baseLogger, id);
```

3. Update route handlers to use `req.logger` when available, falling back to service logger. This is additive — existing `logger.info()` calls in services still work, they just won't have requestId. Route handlers can use `req.logger.info()` for correlated logs.

Do NOT refactor all service methods to accept requestId — that's too invasive for a bug fix sprint. The route-level logging with req.logger is sufficient for request tracing. Services keep their own loggers.

Do NOT change the createLogger() signature — it would break all existing logger instantiations.</action>
