**Bug 1 — Transaction webhook `.success` check (plaid.js:406-426):**
- `handleTransactionWebhook` calls `plaidService.getTransactions()` which returns `{ transactions, totalTransactions, accounts, requestId }`
- Code then checks `if (!transactionsResponse.success)` — but `.success` field NEVER exists on this response
- Result: `!undefined` is truthy → ALWAYS takes the error branch → transactions are NEVER persisted
- This is the most impactful bug: webhook transaction sync is completely broken

**Bug 2 — Webhook handlers return void, no status (plaid.js:382-642):**
- `handleTransactionWebhook`, `handleItemWebhook`, `handleAuthWebhook` return nothing
- The main webhook route handler can't distinguish success from failure
- The 200 response is always sent regardless of handler outcome

**Bug 3 — Webhook always refetches 30 days of transactions (plaid.js:401-410):**
- Fetches 30 days on every webhook instead of using Plaid's `new_transactions` count
- Inefficient but not broken — defer to a future optimization phase

**Established patterns:**
- Plaid routes use Winston child logger
- plaidDataService handles Supabase persistence
- Webhook handlers are async functions called from the main route
