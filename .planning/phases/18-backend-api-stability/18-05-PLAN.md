**Bug 1 — documentPipelineService pipelineState Map never cleaned (documentPipelineService.js:64-65):**
- `this.pipelineState = new Map()` accumulates entries for every processed document
- `initPipeline()` adds entries, but no method removes them after completion
- Impact: Unbounded memory growth — 10K documents = 10K+ Map entries, each holding full pipeline state
- Fix: Add cleanup when pipeline reaches terminal state (completed/failed)

**Bug 2 — Mock Maps in caseFileService and documentService never cleaned:**
- `this.mockCases = new Map()`, `this.mockDocuments = new Map()`, `this.mockDocCaseMap = new Map()`
- These grow during development/testing with no cleanup mechanism
- Impact: Memory leak in development mode; less critical than production pipeline leak
- Fix: Add cleanup methods and/or TTL-based expiry

**Established patterns:**
- Services are singletons (`module.exports = new ServiceClass()`)
- Pipeline has state machine with defined states (PENDING, PROCESSING, COMPLETED, FAILED)
- Winston logging for state changes

  <action>Add automatic cleanup for the pipelineState Map:

1. Find the method that transitions pipeline to terminal states (COMPLETED, FAILED, ERROR). This is likely `updatePipelineState()` or similar.

2. When a pipeline reaches a terminal state, schedule cleanup after a grace period:
```javascript
if (['completed', 'failed', 'error'].includes(newState)) {
  // Keep for 5 minutes so clients can poll final status, then clean up
  setTimeout(() => {
    this.pipelineState.delete(documentId);
  }, 5 * 60 * 1000);
}
```

3. Add a `getMapSize()` method for monitoring: `return this.pipelineState.size`

4. Add a safety valve: if Map exceeds 1000 entries, log a warning and evict the oldest 100 entries (by timestamp if available, or by insertion order since Map maintains order).

Do NOT use a separate interval timer for cleanup — it adds complexity. The terminal state transition is the natural cleanup trigger.

Do NOT delete immediately on terminal state — clients may still be polling for final status.</action>

```javascript
}
```

Do NOT add TTL-based expiry — it adds unnecessary complexity for mock data. Simple size-based eviction is sufficient.</action>
