Output: Updated ocrService.js with quality-aware extraction and tests proving improved reliability.

<task type="auto">
  <name>Task 1: Raise text threshold and add extraction quality heuristics</name>
  <files>backend-express/services/ocrService.js</files>
  <action>
1. Raise MEANINGFUL_TEXT_THRESHOLD from 50 to 200 characters — 50 chars is barely a sentence and causes scanned PDFs with minimal embedded metadata to be treated as text-based, producing garbage extraction.

2. Add a _assessTextQuality(text) helper method that evaluates extracted text quality:
   - wordCount: split on whitespace, count tokens
   - avgWordLength: average characters per word (gibberish detection — avg < 2 or > 20 is suspicious)
   - alphaRatio: ratio of alphabetic characters to total (< 0.3 suggests OCR artifacts or binary)
   - lineCount: number of non-empty lines
   - Return an object: { wordCount, avgWordLength, alphaRatio, lineCount, qualityScore }
   - qualityScore formula: start at 1.0, subtract 0.2 if wordCount < 20, subtract 0.2 if avgWordLength outside 3-15 range, subtract 0.2 if alphaRatio < 0.5, subtract 0.1 if lineCount < 3. Clamp to [0.0, 1.0].

3. In extractFromPdf(): after pdf-parse extraction, if text length >= MEANINGFUL_TEXT_THRESHOLD, call _assessTextQuality(). If qualityScore < 0.4, log a warning and fall through to Claude Vision instead of returning the low-quality text.

4. Log quality metrics at info level for both strategies: { method, charCount, wordCount, qualityScore }.
  </action>
  <verify>cd backend-express && npm test -- --testPathPattern=ocr 2>&1 | tail -20</verify>
  <done>MEANINGFUL_TEXT_THRESHOLD is 200, _assessTextQuality exists and is called, low-quality pdf-parse results fall through to Vision, all existing OCR tests still pass</done>
</task>

<task type="auto">
  <name>Task 2: Replace hardcoded confidence with dynamic quality-based scoring</name>
  <files>backend-express/services/ocrService.js, backend-express/__tests__/services/ocrService.test.js</files>
  <action>
1. Replace the hardcoded confidence values (0.95 for pdf-parse at line ~133, 0.85 for Vision at line ~205) with dynamic confidence based on _assessTextQuality():
   - pdf-parse base: 0.90, multiplied by qualityScore. So excellent text (1.0) = 0.90, mediocre (0.6) = 0.54.
   - Vision base: 0.80, multiplied by qualityScore. So excellent (1.0) = 0.80, mediocre (0.6) = 0.48.
   - Rationale: pdf-parse has higher base because it reads embedded text directly; Vision interprets images which is inherently less reliable. Quality multiplier ensures bad extractions don't carry false confidence.

2. Include qualityMetrics in the returned extraction result object alongside text, confidence, method. This makes quality data available to downstream services.

3. Add/update tests in ocrService.test.js:
   - Test _assessTextQuality with good text (proper sentences) → qualityScore >= 0.8
   - Test _assessTextQuality with gibberish (random chars) → qualityScore < 0.4
   - Test _assessTextQuality with short text (< 20 words) → qualityScore reduced
   - Test that pdf-parse confidence is dynamic (not always 0.95)
   - Test that Vision confidence is dynamic (not always 0.85)
   - Test that qualityMetrics is present in extraction results
   - Ensure all existing tests updated to handle dynamic confidence (use expect.any(Number) or ranges where hardcoded 0.95/0.85 was asserted)
  </action>

  <done>No hardcoded confidence values in ocrService.js, confidence varies by extraction quality, qualityMetrics included in results, all tests pass including new quality scoring tests</done>
