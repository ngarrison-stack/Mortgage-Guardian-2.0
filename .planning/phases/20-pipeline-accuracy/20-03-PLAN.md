<task type="auto">
  <name>Task 1: Add word boundary matching for keyword patterns</name>
  <files>backend-express/services/complianceRuleEngine.js</files>
  <action>
1. Find the method that matches keywords against document/anomaly text (the keyword matching logic in the rule evaluation flow). Currently it likely uses simple string.includes() or indexOf().

2. Replace simple substring matching with word-boundary-aware matching:
   - Create a helper: _matchKeyword(text, keyword) that uses RegExp with word boundaries: new RegExp(`\\b${escapeRegex(keyword)}\\b`, 'i')
   - Add escapeRegex(str) helper to escape special regex characters in keywords (some keywords like "2-1" contain regex-special chars)
   - This ensures "escrow" matches "escrow account" and "Escrow Analysis" but NOT "escrow'd" or "escrowed" (unless "escrowed" is a separate keyword)
   - Case-insensitive matching (the 'i' flag) maintains existing behavior where "ESCROW" matches "escrow"

3. Apply _matchKeyword() everywhere keywords are matched against text in the rule evaluation. Do NOT change the keyword lists themselves — only how they're matched.

4. For field pattern matching (fieldPatterns like 'escrow*'), convert wildcards to proper regex with word boundary at start:
   - 'escrow*' → /^escrow/i (matches field names starting with "escrow")
   - '*Balance' → /Balance$/i (matches field names ending with "Balance")
   - Exact match (no wildcard): use strict equality
   - This prevents 'escrow*' from matching inside longer unrelated strings

5. Log at debug level when keyword/pattern matching changes behavior (matched with old logic but not new, or vice versa) — this helps calibrate in testing. Use logger.debug with { ruleId, keyword, text: text.substring(0, 100) }.
  </action>

  <done>Keyword matching uses word boundaries, field patterns use proper regex, no simple substring matching for keywords, all existing rule engine tests pass</done>
</task>
