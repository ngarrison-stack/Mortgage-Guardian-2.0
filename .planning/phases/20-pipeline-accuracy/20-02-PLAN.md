<task type="auto">
  <name>Task 1: Add confidence threshold and low-confidence handling</name>
  <files>backend-express/services/classificationService.js, backend-express/services/documentPipelineService.js</files>
  <action>
1. In classificationService.js, add constants at the top:
   - CONFIDENCE_HIGH_THRESHOLD = 0.7 (classification trusted, proceed normally)
   - CONFIDENCE_LOW_THRESHOLD = 0.4 (classification unreliable, flag for review)
   - Below 0.4: classification result includes `confidenceLevel: 'low'`
   - 0.4 to 0.7: includes `confidenceLevel: 'medium'`
   - Above 0.7: includes `confidenceLevel: 'high'`

2. In the classifyDocument() return object, add the confidenceLevel field based on the parsed confidence score. This is additive — existing fields (documentType, subType, confidence, category, reasoning) remain unchanged.

3. In documentPipelineService.js _runClassification() method: after classification completes, check confidenceLevel. If 'low':
   - Log a warning: "Low classification confidence ({confidence}), document may be misclassified"
   - Add a `classificationWarning` field to the pipeline state: "Low confidence classification — findings should be reviewed"
   - Do NOT block the pipeline — the document still proceeds, but downstream consumers (analysis, compliance) can see the warning
   - Rationale: Blocking would break the flow for all low-confidence docs. Flagging lets the reporting layer communicate uncertainty.

4. If confidenceLevel is 'medium', log at info level: "Medium classification confidence ({confidence}), proceeding with classification"
  </action>

  <done>confidenceLevel field present in all classification results, pipeline logs warnings for low confidence, classificationWarning field set for low-confidence docs, existing tests pass</done>
</task>

<task type="auto">
  <name>Task 2: Fix user-provided type hint handling and add tests</name>
  <files>backend-express/services/classificationService.js, backend-express/__tests__/services/classificationService.test.js</files>
  <action>
1. In classificationService.js, fix the prompt construction where user-provided existingType is included. Currently the prompt says "classify independently" but also provides the user hint, creating conflicting instructions. Change the prompt to:
   - Remove "classify independently" when a hint is provided
   - Instead: "The uploader suggested this is a '{existingType}' document. Consider this as a starting point but override if the content clearly indicates a different type. If you agree with the suggestion, your confidence should reflect the document content, not the suggestion."
   - When NO hint provided: keep the original "classify independently" instruction
   - Rationale: The current conflicting instruction causes Claude to sometimes ignore obvious hints (lowering accuracy) and sometimes blindly follow bad hints (also lowering accuracy).

2. Add/update tests in classificationService.test.js:
   - Test that confidenceLevel is 'high' when confidence > 0.7
   - Test that confidenceLevel is 'medium' when confidence between 0.4 and 0.7
   - Test that confidenceLevel is 'low' when confidence < 0.4
   - Test that prompt includes hint language when existingType provided
   - Test that prompt includes "classify independently" when no existingType
   - Test that existing classification response parsing still works (regression)
   - Update any existing assertions that check exact prompt text to handle the new variants
  </action>
  <verify>cd backend-express && npm test -- --testPathPattern=classificationService --verbose 2>&1 | tail -30</verify>
  <done>No conflicting "classify independently" + hint in same prompt, confidenceLevel tests pass, all existing classification tests pass</done>
