  <name>Task 1: Integrate encryption into documentService upload and download</name>
  <files>backend-express/services/documentService.js, backend-express/__tests__/services/documentService.test.js</files>
  <action>
**Modify documentService.js:**

1. Add require at top: `const documentEncryptionService = require('./documentEncryptionService');`

2. In `uploadDocument()`:
   - AFTER creating fileBuffer from base64 content: `const fileBuffer = Buffer.from(content, 'base64');`
   - ADD encryption step: check if `process.env.DOCUMENT_ENCRYPTION_KEY` is set
   - If set: `const encryptedBuffer = documentEncryptionService.encrypt(userId, fileBuffer);` then upload encryptedBuffer instead of fileBuffer
   - If not set: log warning once and upload plaintext (graceful degradation for dev environments)
   - Add `encrypted: true/false` to the metadata stored in the documents table insert (add to the insert object)

3. In `getDocument()`:
   - AFTER downloading file from storage and converting to buffer
   - Check if the document metadata has `encrypted: true`
   - If encrypted: `const decryptedBuffer = documentEncryptionService.decrypt(userId, buffer);` then convert decryptedBuffer to base64
   - If not encrypted: convert buffer to base64 as-is (backward compatibility with pre-encryption documents)
   - This ensures old unencrypted documents still download correctly

4. In `deleteDocument()`:
   - No encryption changes needed (just deletes the file, doesn't read content)

**Do NOT encrypt in mock methods** — mock mode is for development without Supabase. Keep mock methods unchanged.

**Update existing tests** in documentService.test.js:
- Mock documentEncryptionService in tests that test Supabase mode
- Verify encrypt is called during upload when DOCUMENT_ENCRYPTION_KEY is set
- Verify decrypt is called during download when document metadata has encrypted: true
- Verify plaintext upload when DOCUMENT_ENCRYPTION_KEY is not set
- Verify backward-compatible download of unencrypted documents

**Review documentPipelineService.js** to understand how it interacts with document storage:

The pipeline processes documents through: upload → OCR → classify → analyze. The OCR step extracts text from the document buffer. If documents are now encrypted at rest, we need to ensure:

1. **During pipeline processing**, the document buffer is in plaintext (OCR needs readable content). The pipeline works with the original buffer BEFORE encryption — encryption happens when documentService stores the final result. So the pipeline itself may not need changes.

2. **Verify the flow:**
   - Pipeline receives fileBuffer (plaintext from client)
   - OCR processes plaintext buffer
   - Classification runs on extracted text
   - documentService.uploadDocument stores the file (encryption happens HERE, inside documentService)
   - Pipeline status tracks processing state

3. **If pipeline stores intermediate results to Supabase Storage** (check the code), those also need encryption. But based on Phase 10-04 design, the pipeline uses in-memory state with best-effort DB persistence for metadata only — file content stays in memory during processing.

4. **Add DOCUMENT_ENCRYPTION_KEY to test environment** setup in the integration test file so encryption path is exercised during integration tests. Or mock the encryption service to verify it's called.

**Update integration tests:**
- Ensure the pipeline integration tests still pass with encryption enabled
- Add a test verifying that the document stored via the pipeline is encrypted (mock documentEncryptionService.encrypt and verify it was called)
- Maintain all existing integration test assertions

**Key principle:** The pipeline processes plaintext. Encryption is a storage concern handled by documentService. The pipeline should NOT encrypt/decrypt — it delegates to documentService which handles encryption transparently.
