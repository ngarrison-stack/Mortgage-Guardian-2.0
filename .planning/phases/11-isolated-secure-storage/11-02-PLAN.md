**Tech stack available:** Supabase Storage, Supabase PostgreSQL (storage.objects table)
**Established patterns:** Storage path `documents/{userId}/{documentId}`, `.from('documents')` bucket access
**Current storage path:** `documents/${userId}/${documentId}` — the `documents/` prefix is redundant with bucket name but harmless; keep for backward compatibility.

**Key insight:** Supabase Storage uses an internal `storage.objects` table with RLS-like policies. The `storage.foldername(name)` function extracts path segments. Since our path is `documents/{userId}/{documentId}`, the userId is at `foldername[2]` (PostgreSQL 1-indexed arrays, first segment is 'documents').

  <action>
Create migration 005 that adds RLS-style policies to the Supabase Storage `storage.objects` table for the 'documents' bucket:

**Policies to create:**

1. **SELECT** — Users can download own documents:
```sql
CREATE POLICY "Users can download own documents" ON storage.objects
  FOR SELECT USING (
    bucket_id = 'documents' AND
    (storage.foldername(name))[2] = auth.uid()::text
  );
```

2. **INSERT** — Users can upload to own folder:
```sql
CREATE POLICY "Users can upload own documents" ON storage.objects
  FOR INSERT WITH CHECK (
    bucket_id = 'documents' AND
    (storage.foldername(name))[2] = auth.uid()::text
  );
```

3. **UPDATE** — Users can update own documents:
```sql
CREATE POLICY "Users can update own documents" ON storage.objects
  FOR UPDATE USING (
    bucket_id = 'documents' AND
    (storage.foldername(name))[2] = auth.uid()::text
  );
```

4. **DELETE** — Users can delete own documents:
```sql
CREATE POLICY "Users can delete own documents" ON storage.objects
  FOR DELETE USING (
    bucket_id = 'documents' AND
    (storage.foldername(name))[2] = auth.uid()::text
  );
```

**Path logic:** Storage path is `documents/{userId}/{documentId}`. `storage.foldername` returns folder segments (excluding filename). With path 'documents/user123/doc456', foldername returns `['documents', 'user123']`. PostgreSQL arrays are 1-indexed, so `[2]` = userId.

**Wrap each policy** in a DO $$ guard checking `pg_policies` for idempotency, matching the pattern from migration 004.

**Note in migration comment:** These policies apply to anon/authenticated key access. The service_role key (used by Express backend) bypasses these policies by default.

3. Call `validateStoragePath` in `getDocument` BEFORE the storage download call
