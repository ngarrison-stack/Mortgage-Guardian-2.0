<task type="auto">
  <name>Task 1: Create encrypted document lifecycle integration tests</name>
  <files>backend-express/__tests__/integration/document-security.test.js</files>
  <action>
Create integration test file that verifies the complete encrypted document lifecycle:

**Test group: "Encrypted Document Lifecycle"** (5-6 tests):

1. **"encrypts document on upload and decrypts on download"**
   - Set DOCUMENT_ENCRYPTION_KEY env var in test setup
   - Mock Supabase storage to capture uploaded buffer
   - Call documentService.uploadDocument with known plaintext content
   - Verify the buffer passed to Supabase storage.upload is NOT the original plaintext (it's encrypted)
   - Verify the buffer starts with 12-byte IV (not recognizable as original file format)
   - Call documentService.getDocument for the same document
   - Mock Supabase storage.download to return the captured encrypted buffer
   - Verify returned content matches original plaintext (round-trip succeeds)

2. **"backward-compatible with unencrypted documents"**
   - Mock getDocument to return document metadata WITHOUT encrypted flag
   - Mock storage.download to return plaintext buffer directly
   - Verify content is returned as-is (no decryption attempted)

3. **"gracefully handles missing encryption key"**
   - Unset DOCUMENT_ENCRYPTION_KEY
   - Call uploadDocument
   - Verify upload succeeds with plaintext (no crash)
   - Verify warning was logged

4. **"different users produce different encrypted outputs for same content"**
   - Encrypt same document content for user-A and user-B
   - Verify encrypted buffers are different (different per-user keys)

5. **"user-A cannot decrypt user-B's document"**
   - Encrypt document as user-A
   - Attempt decrypt as user-B
   - Verify decryption throws error (auth tag mismatch)

6. **"encrypted document survives full pipeline flow"**
   - Mock pipeline dependencies (OCR, classification)
   - Run document through pipeline
   - Verify final stored document is encrypted
   - Verify pipeline status tracking works correctly throughout

Use `beforeEach` to set up fresh encryption key and mocks. Use `afterEach` to clean up env vars.
  </action>
  <verify>cd backend-express && npm test -- --testPathPattern=document-security --verbose</verify>
  <done>6 integration tests passing covering full encrypted lifecycle, backward compatibility, cross-user isolation, and pipeline integration</done>

<task type="auto">
  <name>Task 2: Create cross-user isolation route-level tests</name>
  <files>backend-express/__tests__/integration/user-isolation.test.js</files>
  <action>
Create route-level integration tests verifying cross-user data isolation across all document-related API endpoints:

**Test group: "Cross-User Document Isolation"** (6-8 tests):

1. **"user cannot access another user's documents via GET /v1/documents/:id"**
   - Mock auth middleware for user-A and user-B
   - Create document as user-A via mocked service
   - Request document as user-B via supertest
   - Verify 404 or empty response (not user-A's document)

2. **"user only sees own documents in GET /v1/documents"**
   - Create documents for user-A and user-B
   - List documents as user-A
   - Verify response contains only user-A's documents

3. **"user cannot delete another user's document"**
   - Create document as user-A
   - Attempt delete as user-B via supertest
   - Verify deletion fails (404 or forbidden)

4. **"user cannot access another user's case files via GET /v1/cases/:id"**
   - Create case file as user-A
   - Request case as user-B
   - Verify 404 (not user-A's case)

5. **"user only sees own case files in GET /v1/cases"**
   - Create cases for both users
   - List as user-A
   - Verify only user-A's cases returned

6. **"user cannot modify another user's case file"**
   - Create case as user-A
   - Attempt PUT as user-B
   - Verify update fails

7. **"user cannot associate documents across users"**
   - Create case as user-A, document as user-B
   - Attempt to add user-B's document to user-A's case
   - Verify association fails

Use supertest for route testing. Mock auth middleware to simulate different users (same pattern as `cases-routes.test.js`). Mock services to track which user_id is passed.

**Key principle:** These tests verify the application layer enforces isolation. Combined with RLS (Plan 01), storage policies (Plan 02), and encryption (Plan 03-04), this creates a comprehensive defense-in-depth security posture.
  </action>

  <done>7+ route-level isolation tests passing; cross-user access blocked for documents, cases, and associations; all tests use realistic mocked auth patterns</done>
