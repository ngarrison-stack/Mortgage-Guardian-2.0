<objective>
Decide on and implement malware scanning strategy for uploaded documents.

Purpose: File type validation prevents extension spoofing, but doesn't detect malicious content within valid file types (e.g., a PDF with embedded malware). A malware scanning layer adds defense in depth. However, the serverless deployment target (Vercel/Railway) constrains the options.
Output: Either a malware scanning integration or a documented decision to defer with compensating controls.
</objective>

<task type="checkpoint:decision" gate="blocking">
  <decision>Select malware scanning approach for document uploads</decision>
  <context>
    The upload flow now validates file types via magic numbers and enforces size limits. The remaining gap is detecting malicious content within valid file types (e.g., a PDF exploiting a viewer vulnerability). The deployment target is serverless (Vercel/Railway), which limits infrastructure options.

    Current security stack already provides:
    - JWT authentication on all routes
    - Joi schema validation (type, size, format)
    - Magic number file type verification
    - Filename sanitization
    - Base64 content size limits
  </context>
  <options>
    <option id="virustotal">
      <name>VirusTotal API integration</name>
      <pros>Industry-standard scanning with 70+ AV engines, SaaS (no infrastructure), async scanning possible, well-documented API</pros>
      <cons>Free tier limited (4 requests/min, 500/day), adds external API dependency, async results mean files stored before scan completes, requires API key management</cons>
    </option>
    <option id="defer">
      <name>Defer scanning — document compensating controls</name>
      <pros>No additional complexity or dependencies, current layers already strong (type validation + size limits + auth), can add scanning later without breaking changes, Supabase Storage has its own security policies</pros>
      <cons>No malware detection, relies on file type validation as primary defense, gap in defense-in-depth</cons>
    </option>
    <option id="clamav">
      <name>ClamAV via containerized microservice</name>
      <pros>Open source, self-hosted, no API limits, synchronous scanning</pros>
      <cons>Requires running container (not serverless-compatible), adds infrastructure complexity, needs virus definition updates, overkill for current deployment model</cons>
    </option>
  </options>
  <resume-signal>Select: virustotal, defer, or clamav</resume-signal>
</task>

<task type="auto">
  <name>Task 2: Implement chosen scanning approach</name>
  <files>backend-express/utils/fileValidation.js, backend-express/routes/documents.js</files>
  <action>
    Based on the decision above:

    **If "virustotal":**
    1. Install no additional packages (use native https/fetch for VirusTotal API v3)
    2. Add scanFileContent(buffer) to fileValidation.js:
       - POST buffer to https://www.virustotal.com/api/v3/files
       - Return scan ID for async results
       - Store scan status in document metadata
    3. Add VIRUSTOTAL_API_KEY to .env.example
    4. In upload route: scan after storage (async — don't block upload), store scanId in metadata
    5. Add GET /v1/documents/:documentId/scan-status endpoint to check results

    **If "defer":**
    1. Add a comment block in fileValidation.js documenting the decision:
       - Why deferred (serverless constraints, strong existing validation)
       - What compensating controls exist (magic number, size limits, auth)
       - When to revisit (if moving to containerized deployment, or if handling high-risk documents)
    2. Add a `scanFileContent` stub that returns { scanned: false, reason: 'Malware scanning deferred — see fileValidation.js' }
    3. Export the stub so future integration is a drop-in replacement

    **If "clamav":**
    1. Document ClamAV container setup requirements
    2. Create scanFileContent() that connects to ClamAV daemon via TCP
    3. Add CLAMAV_HOST, CLAMAV_PORT to .env.example
    4. Wire into upload route (synchronous scan before storage)
  </action>
  <verify>
    If virustotal: verify API call structure is correct (mock test)
    If defer: verify stub exported and documented
    If clamav: verify connection logic is correct
  </verify>
  <done>Chosen scanning approach implemented (or deferred with documented rationale and stub for future integration)</done>
</task>
