<task type="auto">
  <name>Task 1: Create federal statute taxonomy configuration</name>
  <files>backend-express/config/federalStatuteTaxonomy.js</files>
  <action>
Create a comprehensive federal statute taxonomy configuration file that defines all major federal mortgage lending laws and their requirements. Structure as a FEDERAL_STATUTES object with entries for each statute:

**Statutes to define:**
1. **RESPA** (Real Estate Settlement Procedures Act) — Sections 6 (QWR), 8 (kickbacks), 10 (escrow)
2. **TILA/Reg Z** (Truth in Lending Act) — disclosure requirements, APR accuracy, right of rescission
3. **ECOA/Reg B** (Equal Credit Opportunity Act) — discrimination, adverse action notices
4. **FDCPA** (Fair Debt Collection Practices Act) — collection practices, validation notices
5. **SCRA** (Servicemembers Civil Relief Act) — 6% interest rate cap, foreclosure protections
6. **HMDA/Reg C** (Home Mortgage Disclosure Act) — reporting requirements
7. **CFPB/Reg X** (Dodd-Frank, CFPB Servicing Rules) — loss mitigation, dual tracking, force-placed insurance

Each statute entry should contain:
- `id`: Short identifier (e.g., 'respa', 'tila')
- `name`: Full name
- `citation`: Primary statutory citation (e.g., '12 U.S.C. § 2601 et seq.')
- `regulatoryBody`: Enforcing agency (e.g., 'CFPB')
- `sections`: Array of section objects, each with:
  - `id`: Section identifier (e.g., 'respa_s6')
  - `section`: Section reference (e.g., 'Section 6')
  - `title`: Section title (e.g., 'Qualified Written Requests')
  - `regulatoryReference`: CFR citation (e.g., '12 CFR § 1024.35-36')
  - `requirements`: Array of specific requirement strings
  - `violationPatterns`: Array of objects mapping to forensic finding types:
    - `discrepancyType`: matches DISCREPANCY_TYPES from crossDocumentAnalysisSchema
    - `anomalyType`: matches ANOMALY_TYPES from analysisReportSchema
    - `keywords`: Array of keyword strings to match in finding descriptions
    - `severity`: Default violation severity for this pattern
  - `penalties`: String describing potential penalties

Export FEDERAL_STATUTES and helper functions:
- `getStatuteById(id)` — lookup by short id
- `getSectionById(sectionId)` — lookup section across all statutes
- `getStatuteIds()` — return all statute ids
- `getSectionIds()` — return all section ids
- `getViolationPatternsForDiscrepancyType(type)` — find all matching sections for a discrepancy type

Use CommonJS exports. Include JSDoc on all exports. This is config data, not business logic — keep it declarative.
  </action>
  <verify>node -e "const t = require('./backend-express/config/federalStatuteTaxonomy'); console.log('Statutes:', t.getStatuteIds().length); console.log('Sections:', t.getSectionIds().length); console.log('RESPA:', t.getStatuteById('respa').name);"</verify>
  <done>7 federal statutes defined with sections, requirements, violation patterns, and penalties. All helper functions return correct results. At least 20 sections total across all statutes.</done>

<task type="auto">
  <name>Task 2: Create compliance report Joi schema</name>
  <files>backend-express/schemas/complianceReportSchema.js</files>
  <action>
Create a Joi validation schema for compliance reports — the output contract between the compliance engine and all downstream consumers.

**Schema structure:**

Top-level `complianceReportSchema`:
- `caseId`: string, required
- `analyzedAt`: ISO-8601 date string, required
- `statutesEvaluated`: array of statute id strings, required
- `violations`: array of violation objects (see below), default []
- `complianceSummary`: summary object (see below), required
- `legalNarrative`: string (Claude-generated legal analysis), optional
- `_metadata`: object (duration, warnings, steps), optional

**Violation object** (`violationSchema`):
- `id`: string (e.g., 'viol-001'), required
- `statuteId`: string (e.g., 'respa'), required
- `sectionId`: string (e.g., 'respa_s6'), required
- `statuteName`: string (full name), required
- `sectionTitle`: string, required
- `citation`: string (e.g., '12 U.S.C. § 2607; 12 CFR § 1024.14'), required
- `severity`: string, valid from SEVERITY_LEVELS ['critical', 'high', 'medium', 'low', 'info'], required
- `description`: string (human-readable), max 2000, required
- `evidence`: array of evidence objects, min 1, required
  - Each evidence: `{ sourceType: 'discrepancy'|'anomaly'|'timeline_violation'|'payment_issue', sourceId: string, description: string }`
- `legalBasis`: string (why this constitutes a violation), max 2000, required
- `potentialPenalties`: string, optional
- `recommendations`: array of strings, default []

**complianceSummary** object:
- `totalViolations`: number, min 0, required
- `criticalViolations`: number, min 0, required
- `highViolations`: number, min 0, required
- `statutesViolated`: array of strings (statute ids with violations), required
- `overallComplianceRisk`: string, valid from RISK_LEVELS ['low', 'medium', 'high', 'critical'], required
- `keyFindings`: array of strings, max 20, default []
- `recommendations`: array of strings, max 20, default []

Export: `complianceReportSchema`, `validateComplianceReport(data)`, `EVIDENCE_SOURCE_TYPES`, `SEVERITY_LEVELS`, `RISK_LEVELS`.

Follow the exact Joi patterns from analysisReportSchema.js and crossDocumentAnalysisSchema.js. Use the same comment section divider style (`// ----------` with JSDoc).
  </action>
  <verify>node -e "const s = require('./backend-express/schemas/complianceReportSchema'); const { error } = s.validateComplianceReport({ caseId: 'test', analyzedAt: new Date().toISOString(), statutesEvaluated: ['respa'], violations: [], complianceSummary: { totalViolations: 0, criticalViolations: 0, highViolations: 0, statutesViolated: [], overallComplianceRisk: 'low', keyFindings: [], recommendations: [] } }); console.log('Valid:', !error);"</verify>
  <done>Schema validates a minimal compliance report. Schema validates a full report with violations and evidence. Schema rejects invalid data (missing required fields, invalid enum values). All exports available.</done>
</task>
