<task type="auto">
  <name>Task 1: Create compliance rule mapping configuration</name>
  <files>backend-express/config/complianceRuleMappings.js</files>
  <action>
Create a mapping configuration that connects forensic analysis findings to federal statute violations. This is the core "rule book" — when the compliance engine encounters a finding, it uses these mappings to determine which statutes may be violated.

**Structure — COMPLIANCE_RULE_MAPPINGS:**

An array of rule objects, each defining:
- `ruleId`: Unique identifier (e.g., 'rule-respa-001')
- `sectionId`: References section from federalStatuteTaxonomy (e.g., 'respa_s6')
- `category`: Grouping ('escrow', 'fees', 'disclosure', 'timing', 'collections', 'servicing', 'discrimination')
- `matchCriteria`: Object defining when this rule fires:
  - `discrepancyTypes`: Array of DISCREPANCY_TYPES that trigger this rule (or empty)
  - `anomalyTypes`: Array of ANOMALY_TYPES that trigger this rule (or empty)
  - `timelineViolation`: boolean — matches timeline violations
  - `paymentIssue`: boolean — matches payment verification issues
  - `keywords`: Array of strings to match in finding descriptions (case-insensitive)
  - `fieldPatterns`: Array of field name patterns to match (e.g., 'escrow*', 'fee*', 'interest*')
  - `minSeverity`: Minimum finding severity to trigger ('info', 'low', 'medium', 'high', 'critical')
- `violationSeverity`: Default severity when this rule fires ('low', 'medium', 'high', 'critical')
- `severityElevation`: Object defining when to elevate severity:
  - `conditions`: Array of conditions (e.g., 'amount > 100', 'repeated', 'critical_field')
  - `elevatedSeverity`: The severity to elevate to
- `descriptionTemplate`: String template for violation description (uses {field}, {amount}, {date} placeholders)
- `legalBasisTemplate`: String template for legal basis explanation

**Define rules for each statute:**

RESPA rules (~8 rules):
- Escrow account violations (Section 10) — escrow discrepancies, analysis errors
- Fee irregularities (Section 8) — kickbacks, unearned fees, fee splitting
- QWR response failures (Section 6) — timing violations, missing correspondence
- Settlement statement errors — amount mismatches in closing docs
- Servicing transfer notification failures — party mismatches, missing notices

TILA/Reg Z rules (~6 rules):
- APR disclosure errors — rate/calculation discrepancies
- Payment amount disclosure errors — amount mismatches
- Right of rescission violations — timing issues
- Finance charge errors — calculation errors
- Adjustable rate notification failures — missing correspondence

ECOA rules (~3 rules):
- Adverse action notice failures — missing correspondence
- Discriminatory terms — term contradictions
- Appraisal copy requirements — missing documents

FDCPA rules (~4 rules):
- Validation notice failures — missing correspondence
- False/misleading representations — amount mismatches
- Unfair collection practices — fee irregularities
- Communication violations — timeline violations

SCRA rules (~3 rules):
- Interest rate cap violations — rate discrepancies above 6%
- Foreclosure protection violations — timeline violations
- Default judgment protections — timing issues

HMDA rules (~2 rules):
- Reporting accuracy — data inconsistencies
- Rate spread reporting — calculation errors

CFPB/Reg X rules (~6 rules):
- Force-placed insurance violations — fee irregularities, missing notices
- Dual tracking violations — timeline violations during loss mitigation
- Loss mitigation timeline failures — timing violations
- Error resolution response failures — QWR-related
- Payment crediting violations — payment date mismatches
- Periodic statement errors — amount/date discrepancies

Total: ~32 rules.

Export: `COMPLIANCE_RULE_MAPPINGS`, `getRulesForSection(sectionId)`, `getRulesForCategory(category)`, `matchRules(finding)` (returns all rules whose matchCriteria are satisfied by the finding object).

The `matchRules(finding)` function should accept an object with: `{ type, severity, description, fields, discrepancyType, anomalyType, isTimelineViolation, isPaymentIssue }` and return matching rules sorted by violationSeverity (highest first).

Create a mapping that defines which federal statutes are most relevant for each document type/subtype from the existing DOCUMENT_TAXONOMY. This enables the compliance engine to prioritize statute evaluation based on the documents present in a case.

**Structure — DOCUMENT_STATUTE_RELEVANCE:**

Object keyed by document category (origination, servicing, correspondence, legal, financial, insurance), each containing subtype mappings:

```
{
  origination: {
    loan_application_1003: {
      primaryStatutes: ['tila', 'ecoa', 'hmda'],
      relevantSections: ['tila_disclosure', 'ecoa_adverse', 'hmda_reporting'],
      complianceFocus: ['disclosure_accuracy', 'fair_lending']
    },
    closing_disclosure: {
      primaryStatutes: ['tila', 'respa'],
      relevantSections: ['tila_disclosure', 'respa_s8', 'respa_s10'],
      complianceFocus: ['fee_accuracy', 'escrow_setup', 'apr_disclosure']
    },
    // ... all origination subtypes
  },
  servicing: {
    monthly_statement: {
      primaryStatutes: ['respa', 'tila', 'cfpb'],
      relevantSections: ['cfpb_periodic_stmt', 'tila_disclosure', 'respa_s10'],
      complianceFocus: ['payment_accuracy', 'escrow_analysis', 'fee_disclosure']
    },
    // ... all servicing subtypes
  },
  // ... correspondence, legal, financial, insurance categories
}
```

Export: `DOCUMENT_STATUTE_RELEVANCE`, `getRelevantStatutes(category, subtype)`, `getRelevantSections(category, subtype)`, `getComplianceFocus(category, subtype)`.

Helper functions should gracefully return defaults (all statutes, empty sections) for unknown subtypes — same graceful degradation pattern as documentFieldDefinitions.js.
