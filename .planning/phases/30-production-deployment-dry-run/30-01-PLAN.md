<task type="auto">
  <name>Task 2: Create deployment validation script</name>
  <files>scripts/validate-deployment.sh</files>
  <action>Create a comprehensive deployment validation script at `scripts/validate-deployment.sh` that validates a running Mortgage Guardian deployment:

  ```bash
  #!/usr/bin/env bash
  # Usage: ./scripts/validate-deployment.sh [BASE_URL]
  # Default: http://localhost:3000
  ```

  The script should test:

  **A. Health endpoints:**
  - GET /health → 200, response contains "status"
  - GET /health/live → 200
  - GET /health/ready → 200 or 503 (503 acceptable if Supabase not configured)

  **B. Security headers (backend):**
  - Response includes X-Content-Type-Options: nosniff
  - Response does NOT include X-Powered-By
  - Response includes Strict-Transport-Security (if HTTPS)

  **C. API structure:**
  - GET /nonexistent-route → 404 with JSON error (no route enumeration in production)
  - GET /api-docs → 404 in production mode (restricted in 29-02)

  **D. Environment validation:**
  - Run `node -e "require('./utils/envValidator').validateEnvironment()"` from backend dir
  - Report which tier of env vars are configured

  **Output format:**
  ```
  Mortgage Guardian Deployment Validation
  ========================================
  Target: http://localhost:3000
  
  [PASS] Health endpoint responds (200)
  [PASS] Liveness probe responds (200)
  [WARN] Readiness probe returns 503 (Supabase not initialized)
  [PASS] X-Powered-By header removed
  [PASS] X-Content-Type-Options: nosniff present
  [PASS] 404 handler returns minimal info
  [PASS] /api-docs not accessible in production
  
  Results: 6 passed, 1 warning, 0 failed
  ```

  Make the script:
  - Accept BASE_URL as first argument (default localhost:3000)
  - Use curl with timeout (5s per request)
  - Exit 0 if all pass/warn, exit 1 if any fail
  - Color-coded output (green PASS, yellow WARN, red FAIL)
  - chmod +x

  Reference the existing `backend-express/test-deployment.sh` for style, but make this one comprehensive and platform-agnostic (not Railway-specific).</action>
  <verify>Script is executable, `bash -n scripts/validate-deployment.sh` passes syntax check</verify>
  <done>Deployment validation script exists, is executable, covers health/security/API checks, documented in DEPLOY.md</done>
</task>

<task type="auto">
  <name>Task 3: Document rollback procedures in DEPLOY.md</name>
  <files>DEPLOY.md</files>
  <action>Add a "Rollback Procedures" section to DEPLOY.md covering rollback for each deployment platform:

  **Docker Compose:**
  - Stop current: `docker compose down`
  - Checkout previous version: `git checkout <previous-tag>`
  - Rebuild and restart: `docker compose up --build -d`

  **Railway:**
  - Use Railway dashboard to redeploy previous deployment
  - Or: `railway rollback` CLI command
  - Database: Supabase point-in-time recovery

  **Vercel:**
  - Use Vercel dashboard → Deployments → Promote previous deployment
  - Or: `vercel rollback` CLI command
  - Instant rollback (previous build artifacts preserved)

  **General Docker:**
  - Tag images before deploying: `docker tag app:latest app:v-previous`
  - Rollback: `docker stop current && docker run app:v-previous`

  Also add a "Pre-Deployment Checklist" section:
  - [ ] All tests pass (`npm test --prefix backend-express`)
  - [ ] Frontend builds (`npm run build --prefix frontend`)
  - [ ] Environment variables configured (run validate-deployment.sh)
  - [ ] Database migrations applied
  - [ ] Previous version tagged for rollback

  Keep additions concise — add to existing sections where possible, don't duplicate.</action>
