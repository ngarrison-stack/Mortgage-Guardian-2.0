<task type="auto">
  <name>Task 1: Create incident response runbook</name>
  <files>RUNBOOK.md</files>
  <action>Create RUNBOOK.md at the project root covering incident response procedures:

  **Structure:**

  1. **Incident Severity Levels**
     - P1 (Critical): Service completely down, data loss risk
     - P2 (High): Major feature broken, degraded for many users
     - P3 (Medium): Minor feature broken, workaround exists
     - P4 (Low): Cosmetic issue, no functional impact

  2. **Immediate Response Checklist** (first 5 minutes)
     - Check health endpoints: `curl /health`, `/health/live`, `/health/ready`
     - Check Sentry for errors (if configured)
     - Check `/metrics` for response time spikes or memory issues
     - Run `./scripts/validate-deployment.sh` against production URL
     - Check container/process status

  3. **Common Incident Playbooks:**
     - **Backend not responding:** Check process, Docker container, port binding, env vars
     - **High response times:** Check /metrics for p95/p99, check Supabase connection, check Redis
     - **Authentication failures:** Check Supabase status, JWT token validation, CORS config
     - **Document processing failures:** Check Claude API key, rate limits, document size limits
     - **Database connection issues:** Check Supabase status, connection string, RLS policies
     - **Memory issues:** Check /metrics heapUsed vs heapTotal, restart if >80% threshold
     - **Plaid integration down:** Check Plaid status page, API keys, webhook config

  4. **Rollback Decision Matrix**
     - When to rollback vs. hotfix
     - Reference DEPLOY.md rollback procedures (don't duplicate)

  5. **Post-Incident**
     - Template for post-mortem notes
     - What to document: timeline, root cause, fix, prevention

  Keep it concise and actionable — operators should be able to follow it under pressure. Use numbered steps, not paragraphs. Reference existing docs (DEPLOY.md, ENV-GUIDE.md) rather than duplicating content.</action>
  <verify>RUNBOOK.md exists, contains severity levels, at least 5 incident playbooks, references DEPLOY.md</verify>
  <done>Incident response runbook covers all major failure modes with actionable steps</done>
</task>

<task type="auto">
  <name>Task 2: Create monitoring and observability guide</name>
  <files>MONITORING.md</files>
  <action>Create MONITORING.md at the project root covering how to monitor and observe Mortgage Guardian:

  **Structure:**

  1. **Health Endpoints**
     - `/health` — Full status with uptime, version, environment, service checks
     - `/health/live` — Liveness probe (is the process running?)
     - `/health/ready` — Readiness probe (are dependencies connected?)
     - Expected responses and what each status means
     - How to configure Kubernetes/Docker health checks using these

  2. **Metrics Endpoint**
     - `/metrics` — Request count, response time percentiles (p50/p95/p99), error rates, memory usage
     - How to interpret each metric
     - Alert thresholds from PERFORMANCE-BASELINE.md (if exists) or Phase 28 targets:
       - p99 response time > 1000ms
       - Error rate > 1%
       - Memory usage > 80% of heap

  3. **Sentry Error Tracking**
     - Setup: Set SENTRY_DSN env var (backend) and NEXT_PUBLIC_SENTRY_DSN (frontend)
     - What gets captured: unhandled exceptions, Express errors
     - What does NOT get captured: PII, request bodies, auth tokens
     - How to triage Sentry alerts

  4. **Log Analysis**
     - Winston structured logging format (JSON in production)
     - Key log fields: requestId, userId, level, message
     - How to search logs for specific requests using requestId
     - Log levels: error, warn, info, debug

  5. **Key Metrics to Watch**
     - Response time trends (gradual increase = potential leak)
     - Error rate spikes (deploy correlation)
     - Memory usage over time (should be stable, not climbing)
     - Health check failures (dependency issues)

  Reference existing endpoints and middleware — don't duplicate implementation details. Focus on "how to use" not "how it works".</action>
