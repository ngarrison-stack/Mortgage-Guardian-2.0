<task type="auto">
  <name>Task 1: Create case file and classification database migration</name>
  <files>backend-express/migrations/002_case_files_and_classifications.sql</files>
  <action>
Create a SQL migration with three changes:

1. **case_files table:**
   - id: uuid PRIMARY KEY DEFAULT gen_random_uuid()
   - user_id: text NOT NULL (matches existing documents.user_id type)
   - case_name: text NOT NULL (e.g., "Smith Mortgage Audit 2024")
   - borrower_name: text
   - property_address: text
   - loan_number: text
   - servicer_name: text
   - status: text NOT NULL DEFAULT 'open' CHECK (status IN ('open', 'in_review', 'complete', 'archived'))
   - notes: text
   - metadata: jsonb DEFAULT '{}'
   - created_at: timestamptz DEFAULT now()
   - updated_at: timestamptz DEFAULT now()
   - INDEX on (user_id, status)
   - INDEX on (user_id, created_at DESC)

2. **document_classifications table:**
   - id: uuid PRIMARY KEY DEFAULT gen_random_uuid()
   - document_id: text NOT NULL (references documents.document_id)
   - classification_type: text NOT NULL (the broad category)
   - classification_subtype: text (the specific document type)
   - confidence: numeric(4,3) CHECK (confidence >= 0 AND confidence <= 1)
   - extracted_metadata: jsonb DEFAULT '{}' (key fields pulled from doc: dates, amounts, parties)
   - classified_by: text NOT NULL DEFAULT 'claude' (could be 'manual' or 'claude')
   - classified_at: timestamptz DEFAULT now()
   - INDEX on (document_id)
   - INDEX on (classification_type)

3. **Alter documents table:** Add columns:
   - case_id: uuid REFERENCES case_files(id) ON DELETE SET NULL
   - classification_id: uuid REFERENCES document_classifications(id) ON DELETE SET NULL
   - INDEX on documents(case_id)

Use IF NOT EXISTS for safety. Add comments on the tables explaining their purpose.

Do NOT use foreign key on document_id to documents table since document_id in the existing documents table may not have a formal PK constraint — use application-level integrity instead.
  </action>

Create `caseFileService.js` following the existing singleton + mock fallback pattern from documentService.js:

**CaseFileService class with methods:**
- `createCase({ userId, caseName, borrowerName, propertyAddress, loanNumber, servicerName, notes })` — Insert into case_files, return created case
- `getCasesByUser({ userId, status, limit = 50, offset = 0 })` — List user's cases, optionally filtered by status, ordered by created_at DESC
- `getCase({ caseId, userId })` — Get single case with its documents (join documents where case_id matches)
- `updateCase({ caseId, userId, updates })` — Update case fields (caseName, borrowerName, propertyAddress, loanNumber, servicerName, status, notes). Only update provided fields.
- `deleteCase({ caseId, userId })` — Delete case (documents remain but case_id set to NULL by FK cascade)
- `addDocumentToCase({ caseId, documentId, userId })` — Update documents.case_id for a document
- `removeDocumentFromCase({ documentId, userId })` — Set documents.case_id to NULL

**Mock methods:** Mirror each method with in-memory Map storage (same pattern as documentService.js mock methods).
