<task type="auto">
  <name>Task 1: Create pipeline timing assertion tests</name>
  <files>backend-express/__tests__/integration/pipeline-performance.test.js</files>
  <action>
Create a test suite that measures execution time of each pipeline step and asserts it completes within generous thresholds. These thresholds are NOT production performance targets — they're canary thresholds designed to catch order-of-magnitude regressions (e.g., a step going from 50ms to 5000ms due to an accidental synchronous loop or N+1 query).

Since all external services are mocked, the thresholds are for internal orchestration logic only:

**Per-step thresholds (with mocked externals):**
- Document pipeline (processDocument): < 500ms per document
- Forensic analysis (runAnalysis): < 500ms for a 3-document case
- Compliance evaluation (evaluateCompliance): < 500ms
- Consolidated report (generateReport): < 500ms
- Full pipeline (intake through report): < 2000ms total

**Implementation:**
- Use `performance.now()` or `Date.now()` to measure each orchestrator call
- Use Jest's `expect(duration).toBeLessThan(threshold)` for assertions
- Run each measurement 3 times and use the median to reduce flakiness
- Add a `describe('performance benchmarks')` block with clear test names like "document pipeline completes within 500ms"

**Important:** These are NOT load tests. They run with mocked externals and verify that internal orchestration logic doesn't have performance regressions. The thresholds are generous (10x expected time) to prevent CI flakiness while still catching catastrophic regressions.

Do NOT use Jest's `--detectOpenHandles` or `--forceExit` flags — if handles leak, that's a real problem to fix, not suppress.

<task type="auto">
  <name>Task 2: Create resource leak detection tests</name>
  <files>backend-express/__tests__/integration/pipeline-resource-tracking.test.js</files>
  <action>
Create tests that detect resource leaks by running the pipeline multiple times sequentially and verifying cleanup:

**Memory tracking:**
- Record `process.memoryUsage().heapUsed` before and after 5 sequential full pipeline runs
- Assert heap growth is < 10MB total (generous threshold — mocked services should have near-zero growth)
- Force GC between runs if `global.gc` is available (run with `--expose-gc`), otherwise skip GC and use a more generous threshold

**Pipeline state cleanup:**
- Run documentPipelineService.processDocument() for 5 different documents
- After all complete, verify the pipeline's internal state Map has entries only for completed/active pipelines (no orphaned entries from prior runs that were never cleaned up)
- Call getStatus() for each pipeline and verify states are terminal (analyzed/complete/failed)

**Mock call count verification:**
- After a full pipeline run, verify mock call counts match expected:
  - Claude should be called exactly N times (once per classification + once per analysis per document + once per comparison pair + compliance calls)
  - Supabase should not have unclosed query chains
  - Verify mocks are properly reset between runs (no state bleeding)

**Structure:**
- describe('resource tracking') with sub-describes for memory, state, and mock verification
- Use afterAll to log actual measurements for debugging CI failures

Do NOT use external profiling tools. Keep it simple with Node.js built-in `process.memoryUsage()` and internal service state inspection.
