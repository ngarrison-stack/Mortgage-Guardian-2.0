<task type="auto">
  <name>Task 2: Create unified DEPLOY.md deployment guide</name>
  <files>DEPLOY.md</files>
  <action>
Create `DEPLOY.md` at the project root with these sections:

1. **Prerequisites** — Docker, Node.js 20+, required API keys (Anthropic, Plaid, Supabase, Clerk)

2. **Local Development with Docker Compose**:
   - Copy .env.example files: `cp backend-express/.env.example backend-express/.env` and `cp frontend/.env.example frontend/.env`
   - Fill in API keys in both .env files
   - `docker compose up --build`
   - Access: backend at http://localhost:3000, frontend at http://localhost:3001
   - Rebuild after code changes: `docker compose up --build`
   - Stop: `docker compose down`
   - Note: For hot-reload during development, run services locally without Docker (see section 3)

3. **Local Development without Docker**:
   - Install deps: `cd backend-express && npm install` + `cd frontend && npm install`
   - Start backend: `cd backend-express && npm run dev`
   - Start frontend: `cd frontend && npm run dev`
   - Benefits: hot-reload, faster iteration

4. **Railway Deployment** (containerized):
   - Railway auto-detects Dockerfile in each service directory
   - Backend: `cd backend-express && railway login && railway init && railway up`
   - Set env vars via Railway dashboard (reference ENV-GUIDE.md for full list)
   - Frontend: `cd frontend && railway login && railway init && railway up`
   - Pass NEXT_PUBLIC_ vars as Railway env vars — they're baked in at build time
   - Health check already configured in railway.toml

5. **Vercel Deployment** (serverless):
   - Backend: `cd backend-express && vercel --prod` (uses vercel.json)
   - Frontend: `cd frontend && vercel --prod`
   - Set env vars in Vercel dashboard
   - Note: Uses serverless functions, not Docker

6. **Generic Docker Host**:
   - Build backend: `docker build -t mg-backend ./backend-express`
   - Build frontend with real values: `docker build -t mg-frontend --build-arg NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_xxx --build-arg NEXT_PUBLIC_API_URL=https://api.example.com ./frontend`
   - Run backend: `docker run -p 3000:3000 --env-file backend-express/.env mg-backend`
   - Run frontend: `docker run -p 3001:3001 mg-frontend`
   - IMPORTANT: NEXT_PUBLIC_ vars are baked at build time — must be provided as --build-arg

7. **Environment Variables** — Link to ENV-GUIDE.md for complete reference. Brief table of critical vars only (ANTHROPIC_API_KEY, PLAID_CLIENT_ID, PLAID_SECRET, SUPABASE_URL, SUPABASE_ANON_KEY, CLERK keys).

8. **Troubleshooting** — Port conflicts (change in .env), Docker build cache (`docker compose build --no-cache`), Clerk key format (must be format-valid pk_test_/pk_live_), env vars not loading (check .env file exists).

Keep it concise and actionable. Reference ENV-GUIDE.md instead of duplicating.
Do NOT include AWS/ECS/Kubernetes — out of scope.
Do NOT include actual secret values.
