  <name>Task 1: Create docker-compose for local development</name>
  <files>docker-compose.yml, .env.docker</files>
  <action>
1. Create `docker-compose.yml` at repo root with these services:

   **backend:**
   - build: ./backend-express
   - ports: 3000:3000
   - env_file: backend-express/.env
   - depends_on: redis, postgres (with condition: service_healthy)
   - healthcheck: wget to /health
   - volumes: mount backend-express source for hot-reload in dev (./backend-express:/app, but NOT node_modules — use anonymous volume)
   - command: npm run dev (override for development)

   **frontend:**
   - build: ./frontend
   - ports: 3001:3001
   - env_file: frontend/.env
   - depends_on: backend (with condition: service_healthy)
   - volumes: mount frontend source for hot-reload (./frontend:/app, anonymous volume for node_modules and .next)
   - command: npm run dev (override for development)

   **redis:**
   - image: redis:7-alpine
   - ports: 6379:6379
   - healthcheck: redis-cli ping
   - volumes: redis-data named volume for persistence

   **postgres:**
   - image: postgres:16-alpine
   - ports: 5432:5432
   - environment: POSTGRES_USER=postgres, POSTGRES_PASSWORD=postgres, POSTGRES_DB=mortgage_guardian
   - healthcheck: pg_isready
   - volumes: postgres-data named volume for persistence

   **Named volumes:** postgres-data, redis-data

   **Network:** Default bridge network is fine — services reference each other by service name.

2. Create `.env.docker` at repo root as a reference template showing docker-compose-specific overrides (e.g., REDIS_URL=redis://redis:6379, SUPABASE_URL pointing to local postgres). Add a comment at top: "# Copy relevant values to backend-express/.env and frontend/.env for docker-compose usage"

   Do NOT create a production docker-compose. Production deployments use Railway/Vercel, not docker-compose.
   Do NOT hardcode secrets — all sensitive values come from .env files.

<task type="auto">
  <name>Task 2: Create deployment guide</name>
  <files>DEPLOY.md</files>
  <action>
Create `DEPLOY.md` at repo root with deployment instructions for each platform:

Structure:
1. **Prerequisites** — Docker, Node.js 20+, npm 9+, platform CLIs (railway, vercel)

2. **Local Development with Docker**
   - `docker compose up` to start all services
   - Accessing services (backend :3000, frontend :3001, Redis :6379, Postgres :5432)
   - Hot-reload behavior (source mounted as volumes)
   - Stopping: `docker compose down` (add `-v` to also remove data volumes)

3. **Local Development without Docker**
   - Install deps: `cd backend-express && npm install` + `cd frontend && npm install`
   - Start backend: `cd backend-express && npm run dev`
   - Start frontend: `cd frontend && npm run dev`
   - External dependencies needed: Redis, PostgreSQL (or use Supabase cloud)

4. **Railway Deployment (Backend)**
   - `railway login && railway init && railway up`
   - Environment variables to set (reference ENV-GUIDE.md)
   - Health check configuration (already in railway.toml)
   - Monitoring: Railway dashboard logs

5. **Vercel Deployment (Frontend)**
   - `cd frontend && vercel --prod`
   - Build settings: Framework = Next.js, Build command = `npm run build`
   - Environment variables to set (reference ENV-GUIDE.md)
   - Note: NEXT_PUBLIC_ vars must be set before build

6. **Docker Deployment (Generic)**
   - Building images: `docker build -t mg-backend ./backend-express` and `docker build -t mg-frontend ./frontend`
   - For frontend: pass NEXT_PUBLIC_ vars as build args: `--build-arg NEXT_PUBLIC_API_URL=https://api.example.com`
   - Running: port mapping, env vars via `-e` or `--env-file`
   - Health check verification

7. **Environment Variable Reference** — Link to ENV-GUIDE.md

Keep it concise and actionable. No theory — just commands and steps.
Do NOT include actual secret values. Use placeholders consistently.
