<objective>
Build the Plaid transaction cross-reference service that matches document-extracted payment data against actual bank transactions — detecting payment discrepancies, misapplied payments, escrow errors, and fee irregularities.

Purpose: This is the "truth verification" layer. Mortgage documents claim certain payments were made/received and certain fees were charged — Plaid bank transaction data provides independent verification. TDD ensures the matching logic correctly handles date tolerance, amount variance, and the many edge cases in payment reconciliation.
Output: Working, tested Plaid cross-reference service with clear input/output contract.
</objective>

<feature>
  <name>Plaid Transaction Cross-Reference Service</name>
  <files>backend-express/services/plaidCrossReferenceService.js, backend-express/__tests__/services/plaidCrossReferenceService.test.js</files>
  <behavior>
    crossReferencePayments(documentPayments, plaidTransactions, options) → matches and reconciles

    Input:
    - documentPayments: [{
        documentId, documentType, documentSubtype,
        date (YYYY-MM-DD), amount (number, positive = payment made),
        description (string — e.g., "Monthly mortgage payment"),
        fieldSource (string — which extractedData field this came from)
      }]
    - plaidTransactions: [{
        transactionId, amount (number — Plaid convention: positive = debit/payment out),
        date (YYYY-MM-DD), name, merchantName, category, pending
      }]
    - options: {
        dateTolerance: number (days, default 5 — payments can post +/- days from due date),
        amountTolerance: number (dollars, default 0.01 — rounding differences),
        escrowDocumentedMonthly: number or null (from escrow analysis docs),
        documentedFees: [{ type, amount, documentId }] or []
      }

    Output: {
      matchedPayments: [{
        documentDate, documentAmount,
        transactionDate, transactionAmount,
        status: "matched" | "close_match" | "mismatch",
        variance: number (absolute difference),
        documentId, transactionId
      }],
      unmatchedDocumentPayments: [{
        date, amount, documentId, description,
        reason: "no_matching_transaction" | "amount_outside_tolerance" | "date_outside_tolerance"
      }],
      unmatchedTransactions: [{
        date, amount, transactionId, name,
        possibleMatch: documentId or null (nearest match that didn't qualify)
      }],
      escrowAnalysis: {
        documentedMonthlyEscrow: number or null,
        actualDisbursements: [{ date, amount, description (from transaction name) }],
        discrepancy: number,
        findings: [string]
      } or null (if no escrow data provided),
      feeAnalysis: {
        documentedFees: [{ type, amount, documentId }],
        transactionFees: [{ type, amount, transactionDate, transactionId }],
        irregularities: [{ description, severity: "critical"|"high"|"medium"|"low"|"info", amount: number or null }]
      } or null (if no fee data provided),
      summary: {
        totalDocumentPayments: number,
        totalPlaidTransactions: number,
        matched: number,
        closeMatches: number,
        unmatched: number,
        paymentVerified: boolean (true if 80%+ matched/close_match)
      }
    }

    Cases:
    1. Perfect match — document payment of $1,500.00 on 2024-01-01 matches Plaid transaction of $1,500.00 on 2024-01-01 → status: "matched", variance: 0
    2. Close match — amounts within tolerance (e.g., $1,500.00 vs $1,500.01) → status: "close_match", variance: 0.01
    3. Date tolerance — document says 2024-01-01, Plaid shows 2024-01-03 (within 5 days) → still matches
    4. Mismatch — document says $1,500, Plaid shows $1,200 (outside tolerance) → status: "mismatch"
    5. Unmatched document payment — no Plaid transaction within date/amount tolerance → unmatchedDocumentPayments
    6. Unmatched Plaid transaction — mortgage-related transaction with no matching document payment → unmatchedTransactions with possibleMatch hint
    7. Empty inputs — empty documentPayments or plaidTransactions → appropriate empty results, no errors
    8. Pending Plaid transactions — excluded from matching (they may change)
    9. Escrow analysis — compare documented monthly escrow against actual escrow-related disbursements (tax, insurance payments identifiable by category)
    10. Fee analysis — compare documented fees against unexpected charges in Plaid transactions
    11. Multiple document payments matching same transaction — pick best match (closest date + amount), leave others unmatched
    12. Custom tolerance options override defaults

    extractPaymentsFromAnalysis(analysisReports) → helper that extracts payment data from analysis reports
    Input: array of analysis report objects (from individual document analysis)
    Output: array of documentPayments in the shape above

    Cases:
    1. Monthly statement with monthlyPayment amount and paymentDueDate → one payment entry
    2. Payment history with multiple payment entries → multiple payment entries
    3. Document without payment-related fields → empty array (no error)
    4. Analysis report with error: true → skip (empty array)
  </behavior>
  <implementation>
    - Singleton class PlaidCrossReferenceService with createLogger('plaid-cross-reference')
    - crossReferencePayments: core matching logic
      1. Filter out pending Plaid transactions
      2. Sort both arrays by date
      3. For each document payment, find best matching Plaid transaction within tolerances
      4. Use greedy matching (best match first, then remove from candidates)
      5. Remaining unmatched on both sides go to respective arrays
    - extractPaymentsFromAnalysis: scan extractedData for payment-related fields
      - Look for: monthlyPayment + paymentDueDate (monthly statements)
      - Look for: payment entries in payment_history subtypes
      - Use field names from documentFieldDefinitions to identify payment fields
    - Escrow analysis: filter Plaid transactions by category (tax, insurance, escrow)
      - Compare sum of escrow disbursements against documented monthly escrow * months
    - Fee analysis: identify fee-like Plaid transactions (late fees, NSF fees, etc.)
      - Compare against documented fees from analysis reports
      - Flag undocumented fees as irregularities
    - No external API calls in this service — it's pure data transformation
    - plaidService.getTransactions() is called by the orchestrator and results passed in
  </implementation>
</feature>
