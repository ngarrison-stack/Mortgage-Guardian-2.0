  <action>
Create the Claude AI-powered cross-document comparison service. Follow the established pattern from documentAnalysisService.js (singleton class, lazy Anthropic client init, structured logging).

**Class: CrossDocumentComparisonService**

**Main method: async compareDocumentPair(docA, docB, comparisonConfig)**
Parameters:
- docA: { documentId, documentType, documentSubtype, extractedData, anomalies, completeness }
- docB: same shape
- comparisonConfig: { pairId, comparisonFields, discrepancyTypes, forensicSignificance } (from COMPARISON_PAIRS)

Returns: {
  pairId,
  documentA: { documentId, documentType, documentSubtype },
  documentB: { documentId, documentType, documentSubtype },
  discrepancies: [{ id, type, severity, description, documentA: {field, value}, documentB: {field, value}, regulation?, forensicNote? }],
  timelineEvents: [{ date, documentId, documentType, event, significance }],
  timelineViolations: [{ description, severity, relatedDocuments, regulation? }],
  comparisonSummary: string (2-3 sentences),
  error?: boolean,
  errorMessage?: string
}

**Prompt architecture:**

System prompt (shared across all comparison types):
```
You are a forensic mortgage document analyst at a major law firm. You are comparing two mortgage-related documents from the same loan file to identify discrepancies, contradictions, and potential regulatory violations.

Your analysis must be thorough, precise, and cite specific field values. When you identify a discrepancy, explain its significance for the borrower and whether it suggests a servicing error, regulatory violation, or potential fraud.

Respond ONLY with valid JSON matching the specified schema.
```

User prompt construction:
1. Include both documents' extractedData (only the comparisonFields categories from config)
2. Include both documents' anomalies (for context on known issues)
3. Include comparison-type-specific instructions via _getComparisonInstructions(comparisonConfig.pairId)
4. Request structured JSON output matching the return schema above

**Type-specific comparison instructions** (via _getComparisonInstructions method):

- "stmt-vs-stmt": Focus on balance progression (does principal decrease appropriately?), payment amount consistency, unexplained fee changes, escrow balance trajectory, late charge patterns. Flag: balance increases without explanation, fees appearing/disappearing between statements.

- "stmt-vs-closing": Focus on interest rate consistency (should match unless ARM), monthly payment vs original terms, loan amount tracking. Flag: rate changes without ARM adjustment notice, payment amount changes without modification.

- "stmt-vs-paymenthistory": Focus on payment application accuracy — do statement balances reflect payments shown in history? Flag: payments shown in history but not credited on statement, misapplied payments (interest vs principal allocation), phantom late fees for on-time payments.

- "stmt-vs-escrow": Focus on escrow balance tracking — does statement escrow match analysis? Monthly escrow amount consistency. Flag: escrow shortage not matching actual disbursements, excessive escrow cushion beyond legal limits.

- "stmt-vs-modification": Focus on modification terms being applied — new payment amount, new rate, new balance. Flag: old payment still being collected after modification date, modification terms not reflected in subsequent statements.

- "closing-vs-note": Focus on core loan terms consistency — amount, rate, term, payment. These must match exactly. Flag: any discrepancy between closing disclosure and promissory note is significant.

- "stmt-vs-armadjust": Focus on rate adjustment timing and amount — does the new rate on statement match the adjustment notice? Flag: rate applied differently than announced, timing mismatch.

- "correspondence-vs-stmt": General comparison — do servicer claims in correspondence match actual records? Flag: disputed amounts not matching statement records.

- "legal-vs-stmt": Focus on amounts claimed in legal proceedings vs actual records — default amount, arrearage, fees. Flag: inflated amounts in legal filings, fees not documented in servicing records.

- Default (unknown pair): Generic forensic comparison focusing on any shared field discrepancies.

**Configuration:**
- Model: claude-sonnet-4-5-20250514 (same as individual analysis — fast, cost-effective for structured output)
- Max tokens: 4096 (comparison output is moderate-sized)
- Temperature: 0.1 (deterministic for forensic accuracy)

**Error handling (follow documentAnalysisService patterns):**
- Anthropic API errors: return { error: true, errorMessage, pairId }
- JSON parse failures: try markdown code fence extraction, then return rawResponse
- Empty/null input: return { error: true, errorMessage: "Missing document data" }
- Timeout: catch and return error object (do not throw)

**Singleton export:** module.exports = new CrossDocumentComparisonService();
