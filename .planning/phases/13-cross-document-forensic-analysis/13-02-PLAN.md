<objective>
Build the document data aggregation service that collects analysis reports for all documents in a case, normalizes extracted data for comparison, and builds typed comparison pairs.

Purpose: This is the data preparation layer — it bridges the gap between individual document analyses (Phase 12) and cross-document comparison (Plan 03). TDD ensures the aggregation logic correctly handles the many edge cases: documents without analysis, partial analysis results, missing fields, and type-based pair matching.
Output: Working, tested aggregation service with clear input/output contract.
</objective>

<feature>
  <name>Cross-Document Data Aggregation Service</name>
  <files>backend-express/services/crossDocumentAggregationService.js, backend-express/__tests__/services/crossDocumentAggregationService.test.js</files>
  <behavior>
    aggregateForCase(caseId, userId) → collects all analyzed documents in a case and builds comparison pairs

    Input: caseId (string), userId (string)
    Output: {
      caseId,
      documents: [{
        documentId,
        documentType (classificationType),
        documentSubtype (classificationSubtype),
        analysisReport (full report object or null),
        extractedData (normalized — the extractedData from analysis report, or empty object),
        anomalies (array from analysis report, or empty array),
        completeness (from analysis report, or null),
        analyzedAt (string or null)
      }],
      comparisonPairs: [{
        pairId (string — from COMPARISON_PAIRS config id),
        docA: { documentId, documentType, documentSubtype },
        docB: { documentId, documentType, documentSubtype },
        comparisonFields (array of strings from config),
        discrepancyTypes (array of strings from config),
        forensicSignificance (string)
      }],
      documentsWithoutAnalysis: [documentId] (array — docs that had no analysis report),
      totalDocuments (number),
      analyzedDocuments (number)
    }

    Cases:
    1. Case with 3 analyzed monthly_statements → 3 stmt-vs-stmt pairs (A-B, A-C, B-C)
    2. Case with 1 monthly_statement + 1 closing_disclosure → 1 stmt-vs-closing pair
    3. Case with 1 document without analysis → included in documentsWithoutAnalysis, excluded from pairs
    4. Case with 0 or 1 analyzed documents → empty comparisonPairs array
    5. Case with no documents → throws error "No documents found in case"
    6. Case not found → throws error "Case not found"
    7. Documents match multiple comparison pair types → all matching pairs included
    8. Bidirectional pair matching works regardless of document order
  </behavior>
  <implementation>
    - Use caseFileService.getCase() to get case with documents
    - For each document: check if analysis_results exists in the document record (from Supabase)
    - If not in Supabase record, try documentPipelineService.getStatus() to get in-memory analysis
    - Build comparison pairs using getComparisonPairsForDocTypes() from crossDocumentComparisons config
    - For N documents of compatible types, generate all unique pairs (N choose 2, filtered by comparison config)
    - Follow singleton service pattern: class CrossDocumentAggregationService { ... } module.exports = new CrossDocumentAggregationService();
    - Use createLogger('cross-document-aggregation') for structured logging
    - Handle Supabase errors gracefully — log and return partial results
  </implementation>
</feature>
