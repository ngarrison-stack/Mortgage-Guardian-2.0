<task type="auto">
  <name>Task 1: Create cross-document analysis report schema</name>
  <files>backend-express/schemas/crossDocumentAnalysisSchema.js</files>
  <action>
Create a Joi validation schema for the cross-document forensic analysis output. Follow the pattern from analysisReportSchema.js (schema + validateX() helper export).

The schema must validate this structure:

**Top-level fields:**
- caseId (string, required)
- analyzedAt (ISO-8601 string, required)
- documentsAnalyzed (number, integer, min 2, required)
- comparisonPairsEvaluated (number, integer, min 0, required)

**discrepancies** (array of objects, default []):
Each discrepancy:
- id (string, required — unique within report, e.g., "disc-001")
- type (string, required — enum: amount_mismatch, date_inconsistency, party_mismatch, term_contradiction, timeline_violation, calculation_error, missing_correspondence, fee_irregularity)
- severity (string, required — enum: critical, high, medium, low, info)
- description (string, required, max 1000)
- documentA (object, required): { documentId (string), documentType (string), documentSubtype (string), field (string), value (alternatives: string, number, null) }
- documentB (object, required): same shape as documentA
- regulation (string, optional — e.g., "RESPA Section 6")
- forensicNote (string, optional, max 2000)

**timeline** (object, required):
- events (array of objects): { date (ISO-8601 date string), documentId (string), documentType (string), event (string, max 500), significance (string: routine, notable, concerning, critical) }
- violations (array of objects, default []): { description (string, max 1000), severity (enum same as discrepancy), relatedDocuments (array of strings, min 1), regulation (string, optional) }

**paymentVerification** (object or null, default null — null when no Plaid data available):
- verified (boolean)
- transactionsAnalyzed (number, integer, min 0)
- dateRange (object): { start (YYYY-MM-DD string), end (YYYY-MM-DD string) }
- matchedPayments (array): { documentDate (string), documentAmount (number), transactionDate (string), transactionAmount (number), status (enum: matched, close_match, mismatch), variance (number, optional) }
- unmatchedDocumentPayments (array): { date (string), amount (number), documentId (string), description (string) }
- unmatchedTransactions (array): { date (string), amount (number), description (string), possibleMatch (string or null) }
- escrowAnalysis (object or null): { documentedMonthlyEscrow (number), actualDisbursements (array: {date, amount, description}), discrepancy (number), findings (array of strings) }
- feeAnalysis (object or null): { documentedFees (array: {type, amount, documentId}), transactionFees (array: {type, amount, transactionDate}), irregularities (array: {description, severity enum, amount (number, optional)}) }

**summary** (object, required):
- totalDiscrepancies (number, integer, min 0)
- criticalFindings (number, integer, min 0)
- highFindings (number, integer, min 0)
- riskLevel (string, enum: low, medium, high, critical)
- keyFindings (array of strings, max 20 items)
- recommendations (array of strings, max 20 items)

Export: crossDocumentAnalysisSchema (the Joi object), validateCrossDocumentAnalysis(data) helper.

Use Joi.alternatives() where values can be string or number. Use .default([]) for arrays. Use .allow(null) where null is valid.

<task type="auto">
  <name>Task 2: Create comparison pair configuration and discrepancy type taxonomy</name>
  <files>backend-express/config/crossDocumentComparisons.js</files>
  <action>
Create a configuration file defining which document types should be compared and what to look for. This drives the comparison engine's behavior.

**COMPARISON_PAIRS** — array of objects defining comparison rules:

Each pair:
- id (string, unique — e.g., "stmt-vs-stmt", "stmt-vs-closing")
- docTypeA (object): { classificationType, classificationSubtype (or "*" for any subtype) }
- docTypeB (object): same shape
- bidirectional (boolean — if true, A-B and B-A are same comparison)
- comparisonFields (array of strings — which extractedData categories to compare: "amounts", "dates", "rates", "parties", "identifiers")
- discrepancyTypes (array of enum strings — which discrepancy types are relevant)
- description (string — human-readable explanation of what this comparison catches)
- forensicSignificance (string: high, medium, low — how important this comparison type is)

Define these comparison pairs (at minimum):

1. **stmt-vs-stmt** — monthly_statement vs monthly_statement: amounts (principalBalance, monthlyPayment, interestRate, escrowBalance, lateCharges), dates (paymentDueDate, statementDate). Catches: balance calculation errors, unexplained fee increases, payment amount changes without modification notice. HIGH significance.

2. **stmt-vs-closing** — monthly_statement vs closing_disclosure: rates (interestRate, apr), amounts (monthlyPayment, loanAmount). Catches: rate changes without ARM adjustment, payment discrepancies vs original terms. HIGH significance.

3. **stmt-vs-paymenthistory** — monthly_statement vs payment_history: amounts (monthlyPayment, lateCharges), dates (paymentDueDate). Catches: misapplied payments, phantom late fees. HIGH significance.

4. **stmt-vs-escrow** — monthly_statement vs escrow_analysis: amounts (escrowBalance, monthlyPayment). Catches: escrow shortage miscalculation, improper escrow accounting. MEDIUM significance.

5. **stmt-vs-modification** — monthly_statement vs loan_modification: amounts (monthlyPayment, interestRate, principalBalance). Catches: modification terms not applied, continued old payment collection. HIGH significance.

6. **closing-vs-note** — closing_disclosure vs promissory_note: amounts (loanAmount, interestRate), terms (loanTerm). Catches: inconsistent loan terms between docs. HIGH significance.

7. **stmt-vs-armadjust** — monthly_statement vs rate_adjustment_notice: rates (interestRate), dates. Catches: rate applied differently than announced. MEDIUM significance.

8. **correspondence-vs-stmt** — any correspondence/* vs monthly_statement: context-dependent. Catches: servicer claims vs actual records. LOW-MEDIUM significance.

9. **legal-vs-stmt** — any legal/* vs monthly_statement: amounts, dates. Catches: foreclosure amount discrepancies, default amount verification. HIGH significance.

**DISCREPANCY_SEVERITY_RULES** — object mapping discrepancy types to default severity, with field-based elevation rules:
- amount_mismatch: default "high", elevate to "critical" if field is in critical tier
- date_inconsistency: default "medium", elevate to "high" if involves payment dates
- party_mismatch: default "medium"
- term_contradiction: default "high"
- timeline_violation: default "high", elevate to "critical" if regulatory deadline involved
- calculation_error: default "critical"
- missing_correspondence: default "medium"
- fee_irregularity: default "high"

Export: COMPARISON_PAIRS, DISCREPANCY_SEVERITY_RULES, getComparisonPairsForDocTypes(typeA, subtypeA, typeB, subtypeB) helper that returns matching pairs.

Helper function should match both specific subtypes and wildcard ("*") entries, and handle bidirectional matching (if pair defines A=stmt/monthly_statement and B=origination/closing_disclosure, it should match when called with those types in either order).
  </action>
