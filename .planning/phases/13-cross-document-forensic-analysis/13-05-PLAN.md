  <action>
Create the orchestrator service that coordinates the full cross-document forensic analysis flow. Follow the pipeline service pattern (sequenced steps, state tracking, graceful degradation).

**Class: ForensicAnalysisService**

**Main method: async analyzeCaseForensics(caseId, userId, options = {})**
Parameters:
- caseId (string, required)
- userId (string, required)
- options: {
    plaidAccessToken (string, optional — if provided, enables Plaid cross-reference),
    transactionDateRange: { start: "YYYY-MM-DD", end: "YYYY-MM-DD" } (optional — defaults to 12 months back from now),
    dateTolerance (number, optional — days, passed to Plaid cross-reference),
    amountTolerance (number, optional — dollars, passed to Plaid cross-reference)
  }

Returns: object conforming to crossDocumentAnalysisSchema with additional metadata:
{
  caseId,
  analyzedAt,
  documentsAnalyzed,
  comparisonPairsEvaluated,
  discrepancies: [...all discrepancies from all pair comparisons],
  timeline: { events: [...all timeline events], violations: [...all violations] },
  paymentVerification: { ...Plaid results } or null,
  summary: { totalDiscrepancies, criticalFindings, highFindings, riskLevel, keyFindings, recommendations },
  // Metadata (not in schema, attached separately):
  _metadata: {
    duration: number (ms),
    steps: {
      aggregation: { status, duration, documentsFound, pairsGenerated },
      comparison: { status, duration, pairsCompared, pairsFailed },
      plaidCrossReference: { status: "completed"|"skipped"|"failed", duration, reason? },
      consolidation: { status, duration }
    },
    warnings: [string]
  }
}

**Orchestration flow:**

Step 1: AGGREGATE
- Call crossDocumentAggregationService.aggregateForCase(caseId, userId)
- If error (case not found, no docs): return early with error status
- If fewer than 2 analyzed documents: return with warning "Insufficient analyzed documents for cross-document comparison"
- Record step metadata

Step 2: COMPARE PAIRS
- For each comparisonPair from aggregation:
  - Call crossDocumentComparisonService.compareDocumentPair(docA, docB, config)
  - Collect all discrepancies, timeline events, and violations
  - If individual comparison fails (error: true), log warning and continue with remaining pairs
- Deduplicate discrepancies (same field + same documents = keep higher severity)
- Assign sequential discrepancy IDs across all pairs ("disc-001", "disc-002", ...)
- Record step metadata (pairs compared, pairs failed)

Step 3: PLAID CROSS-REFERENCE (optional)
- If options.plaidAccessToken provided:
  - Call plaidService.getTransactions with access token and date range
  - Call plaidCrossReferenceService.extractPaymentsFromAnalysis(analysisReports)
  - Call plaidCrossReferenceService.crossReferencePayments(payments, transactions, options)
  - Attach paymentVerification to report
  - If Plaid call fails: set paymentVerification to null, add warning
- If no access token: set paymentVerification to null, note "skipped" in metadata

Step 4: CONSOLIDATE
- Merge all discrepancies from Step 2 into flat array
- Merge all timeline events, sort by date
- Merge all timeline violations, deduplicate
- Calculate summary:
  - totalDiscrepancies = discrepancies.length
  - criticalFindings = count where severity === "critical"
  - highFindings = count where severity === "high"
  - riskLevel = "critical" if any critical findings, "high" if any high findings, "medium" if any medium discrepancies, "low" otherwise
  - keyFindings = top 10 most significant findings (critical first, then high) — extract description
  - recommendations = generate based on finding types (e.g., discrepancy types → specific recommendations)
- Add Plaid findings to keyFindings/recommendations if applicable
- Validate against crossDocumentAnalysisSchema (warnings only, don't reject)
- Record step metadata

**Recommendation generation logic:**
Map discrepancy types to recommendations:
- amount_mismatch → "Request detailed payment application history from servicer"
- date_inconsistency → "Request complete servicing timeline with supporting documentation"
- term_contradiction → "Compare original loan documents against current servicing terms"
- calculation_error → "Request servicer's calculation methodology and verify independently"
- fee_irregularity → "Request itemized fee breakdown and supporting documentation per RESPA Section 6"
- timeline_violation → "Document timeline with certified mail receipts for regulatory complaint"
- missing_correspondence → "Request complete correspondence log per CFPB Regulation X"
- party_mismatch → "Verify chain of title and servicing transfer documentation"
- Plaid unmatched payments → "Request payment posting details for dates where payments were sent but not credited"
Deduplicate recommendations (same recommendation only once even if multiple discrepancies trigger it).

**Best-effort persistence:**
- After consolidation, attempt to store forensic analysis results in Supabase (case_files table or a new forensic_analyses field)
- If Supabase write fails: log warning, continue — results are returned in response regardless

**Error handling:**
- Case not found → return { error: true, errorMessage: "Case not found" }
- No analyzed documents → return partial result with warning
- Individual comparison failures → skip pair, continue, log warning
- Plaid API failure → skip Plaid step, continue, set paymentVerification to null
- Schema validation failure → attach warnings, return result anyway
