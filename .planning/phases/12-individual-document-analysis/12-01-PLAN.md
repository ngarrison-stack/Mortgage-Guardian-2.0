  <files>backend-express/schemas/analysisReportSchema.js</files>
  <action>
Create a Joi schema that defines the structured output format for document analysis reports. This schema validates what Claude returns and what gets stored in `analysis_results` JSONB columns.

**Schema structure:**

```
analysisReport:
  documentInfo:
    documentType: string (required) — classification type (e.g., "servicing")
    documentSubtype: string (required) — classification subtype (e.g., "monthly_statement")
    analyzedAt: ISO date string (required)
    modelUsed: string (required)
    confidence: number 0-1 (required)

  extractedData:
    dates: object — key-value pairs of named dates
      (e.g., { statementDate: "2024-01-15", paymentDueDate: "2024-02-01", maturityDate: "2054-01-01" })
    amounts: object — key-value pairs of named dollar amounts
      (e.g., { principalBalance: 245000, monthlyPayment: 1523.47, escrowBalance: 3421.00 })
    rates: object — key-value pairs of named percentage rates
      (e.g., { interestRate: 6.5, apr: 6.82 })
    parties: object — key-value pairs of named parties
      (e.g., { borrower: "John Smith", lender: "Wells Fargo", servicer: "Wells Fargo Home Mortgage" })
    identifiers: object — account numbers, loan numbers, property addresses
      (e.g., { loanNumber: "****1234", propertyAddress: "123 Main St" })
    terms: object — key-value pairs of loan/document terms
      (e.g., { loanType: "fixed", amortizationType: "30-year", lienPosition: "first" })
    custom: object — type-specific fields that don't fit above categories

  anomalies: array of objects
    each:
      field: string (required) — which extracted field is anomalous
      type: string (required) — enum: unusual_value, inconsistency, missing_required, calculation_error, regulatory_concern
      severity: string (required) — enum: critical, high, medium, low, info
      description: string (required) — human-readable explanation
      expectedValue: any (optional) — what was expected
      actualValue: any (optional) — what was found
      regulation: string (optional) — relevant regulation if applicable (e.g., "RESPA Section 6")

  completeness:
    score: number 0-100 (required) — percentage of expected fields present
    totalExpectedFields: number (required)
    presentFields: array of strings (required) — field names that were found
    missingFields: array of strings (required) — field names that were NOT found
    missingCritical: array of strings (required) — subset of missingFields that are critical

  summary:
    overview: string (required) — 2-3 sentence analysis summary
    keyFindings: array of strings (required) — top findings
    riskLevel: string (required) — enum: low, medium, high, critical
    recommendations: array of strings (required) — suggested actions
```

**Implementation details:**
- Use Joi.object() with .keys() for each section
- All top-level sections required
- extractedData sub-objects use Joi.object().pattern() for flexible key-value pairs
- anomalies uses Joi.array().items() with enum validation on type and severity
- Export both the schema and a `validateAnalysisReport(data)` helper function that returns { value, error }
- Include JSDoc comments documenting each section's purpose

**What to avoid:** Don't make the schema too rigid for extractedData fields — different document types will have different field names. Use Joi.object().pattern(Joi.string(), Joi.alternatives().try(Joi.string(), Joi.number(), Joi.boolean(), Joi.allow(null))) for flexible extraction.

**Cover ALL 54 subtypes** from classificationService.js DOCUMENT_TAXONOMY. For each subtype define:
- `critical`: Fields essential for document validity (missing = high-severity anomaly)
- `expected`: Fields normally present (missing = medium-severity finding)
- `optional`: Fields sometimes present (missing = info-level note)

**Also export helper functions:**
- `getFieldDefinition(classificationType, classificationSubtype)` — returns field definition or generic fallback
- `getExpectedFieldCount(classificationType, classificationSubtype)` — returns critical.length + expected.length
- `categorizeField(fieldName, classificationType, classificationSubtype)` — returns 'critical', 'expected', 'optional', or 'unknown'

**Domain knowledge to apply:**
- Mortgage servicing: payment amounts, escrow, interest rates, fees are critical
- Origination: loan terms, closing costs, parties are critical
- Correspondence: dates, deadlines, amounts due, borrower rights are critical
- Legal: recording information, parties, legal descriptions are critical
- Financial: income/asset verification amounts and dates are critical
- Regulatory: citation references, disclosure dates, required language are critical

**What to avoid:** Don't try to enumerate every possible field — keep critical/expected lists focused on the most important 5-10 fields per subtype. Optional can be broader. If uncertain about a subtype, provide a reasonable set and document it for future refinement.
