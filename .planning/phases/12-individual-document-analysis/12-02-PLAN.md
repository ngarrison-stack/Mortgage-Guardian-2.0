**Prompt engineering strategy:**

Each type-specific prompt should:
1. Start with the document type context (what this document type IS and what it contains)
2. Provide a structured extraction template (JSON format) tailored to this type
3. Include anomaly detection instructions specific to this document type:
   - Servicing: check payment math, escrow calculations, fee reasonableness
   - Origination: verify loan terms consistency, closing cost itemization
   - Correspondence: check deadline calculations, required notices, borrower rights
   - Legal: verify parties, recording info, legal descriptions
   - Financial: check income/asset consistency, verify totals
   - Regulatory: check required disclosures, timing requirements
4. Ask for a risk assessment (low/medium/high/critical)
5. Request the response as valid JSON matching the schema structure

**Example servicing prompt pattern:**
```
You are a forensic mortgage analyst reviewing a {subtype} document.

Extract ALL data points from this document into the following JSON structure:
{
  "extractedData": {
    "dates": { /* named dates found in document */ },
    "amounts": { /* named dollar amounts */ },
    "rates": { /* named percentage rates */ },
    "parties": { /* named parties */ },
    "identifiers": { /* account/loan numbers, addresses */ },
    "terms": { /* loan/document terms */ }
  },
  "anomalies": [
    {
      "field": "fieldName",
      "type": "unusual_value|inconsistency|missing_required|calculation_error|regulatory_concern",
      "severity": "critical|high|medium|low|info",
      "description": "explanation",
      "expectedValue": "if applicable",
      "actualValue": "if applicable"
    }
  ],
  "summary": {
    "overview": "2-3 sentence summary",
    "keyFindings": ["finding1", "finding2"],
    "riskLevel": "low|medium|high|critical",
    "recommendations": ["action1", "action2"]
  }
}

For a monthly_statement specifically, check:
- Payment amount matches principal + interest + escrow
- Interest calculation matches rate × balance / 12
- Escrow disbursements match expected taxes/insurance
- Late fees comply with RESPA limits (typically 4-5% of P&I)
- Fees are itemized and reasonable

Document text:
---
{documentText}
---

Respond ONLY with valid JSON. No markdown, no explanation outside the JSON.
```

**Claude API configuration:**
- Model: `claude-sonnet-4-5-20250514` (consistent with classification service)
- Temperature: 0.1 (deterministic extraction)
- Max tokens: 8192 (analysis reports can be substantial — larger than classification)

**Error handling:**
- Claude API error: log error, return { error: true, errorMessage, rawResponse: null }
- JSON parse failure: return partial result with { parseError: "message", rawResponse: "claude's text" }
- Schema validation failure: log warning, return result with { validationWarnings: [...] } but don't reject
- Missing document text: throw immediately with descriptive error

**Logging:**
- Log analysis start/completion with timing
- Log model used and token counts (from Claude response usage)
- Log anomaly count and risk level for monitoring
- Use existing Winston logger pattern

**What to avoid:**
- Don't import or reference documentEncryptionService (encryption is storage concern, not analysis)
- Don't persist results to database (that's the pipeline's job)
- Don't hard-code field expectations in prompts (use documentFieldDefinitions.js)
- Don't use `jsonwebtoken` or auth logic (this is a pure analysis service)
