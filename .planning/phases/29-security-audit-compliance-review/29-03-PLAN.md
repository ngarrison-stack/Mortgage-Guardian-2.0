<task type="auto">
  <name>Task 1: Security configuration hardening</name>
  <files>backend-express/server.js, frontend/next.config.ts</files>
  <action>Apply security hardening measures:

  **A. Helmet.js configuration review (backend):**
  - Review current Helmet defaults. Ensure these are active:
    - `contentSecurityPolicy`: Verify it doesn't break the Swagger UI (/api-docs). If CSP blocks Swagger assets, add specific exceptions for /api-docs route only.
    - `hsts`: Verify maxAge is at least 1 year (31536000), includeSubDomains: true
    - `referrerPolicy`: Should be 'strict-origin-when-cross-origin' or 'no-referrer'
  - If Helmet is using all defaults with no options object, this is fine — Helmet 8.x defaults are secure. Only add explicit config if a specific setting needs changing.

  **B. CORS hardening:**
  - Add a comment in server.js documenting that ALLOWED_ORIGINS should be explicitly set in production
  - The existing wildcard warning log is good — no code change needed unless the audit found issues

  **C. Frontend security headers (Next.js):**
  - Check `frontend/next.config.ts` for security headers configuration
  - If not present, add `headers()` config for: X-Frame-Options DENY, X-Content-Type-Options nosniff, Referrer-Policy strict-origin-when-cross-origin
  - Next.js serves these via the config, not Helmet

  **D. API documentation access control:**
  - If /api-docs (Swagger UI) should be restricted in production, add NODE_ENV check to only mount it in development
  - If it's intentionally public (for API consumers), document this decision

  Do NOT over-engineer — only apply hardening that addresses real risks for this deployment model.</action>
  <verify>npm test --prefix backend-express passes, npm run build --prefix frontend succeeds. Security headers verified in response.</verify>
  <done>Helmet config reviewed/hardened, CORS documented, frontend headers configured, API docs access policy decided and implemented</done>
