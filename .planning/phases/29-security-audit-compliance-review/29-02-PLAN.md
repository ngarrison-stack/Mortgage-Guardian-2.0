**Known issues from codebase analysis:**
- 404 handler leaks full route list (information disclosure)
- Error handler leaks stack traces in non-production (acceptable but verify production behavior)
- CORS configured with wildcard warning but no enforcement
- No CSRF protection (stateless JWT API — verify this is sufficient)
- Malware scanning stub (documented deferral from Phase 4)

**Established security layers:**
- Helmet.js security headers
- JWT auth on all /v1/ routes
- Joi validation on all 13 endpoints (stripUnknown: true)
- File type validation via magic numbers
- AES-256-GCM document encryption with HKDF per-user keys
- Rate limiting (100 req/15min on /v1/)
- RLS policies in Supabase (user_id isolation)

<task type="auto">
  <name>Task 1: Fix information disclosure vulnerabilities</name>
  <files>backend-express/server.js, backend-express/middleware/errorHandler.js</files>
  <action>Fix the following information disclosure issues:

  **A. 404 handler route enumeration (server.js):**
  The 404 handler returns `availableRoutes` array listing all API endpoints. This is useful for development but leaks the API surface in production.
  - In production (NODE_ENV=production): Return only `{ error: 'Not Found', message: 'Route not found' }` — no route list, no method/path echo
  - In development: Keep the current verbose response with route list for DX
  
  **B. Error handler stack trace leakage (server.js):**
  The global error handler already checks NODE_ENV for stack traces — verify this is correct. Also ensure:
  - Production: Generic "Internal server error" message, no stack
  - Development: Full error details + stack
  - Never expose internal service names, database errors, or file paths in production error messages
  
  **C. Sensitive header removal:**
  Verify Helmet.js is hiding the X-Powered-By header (Express default). Check that no custom headers leak server info.
  
  Do NOT remove development-mode debug info — only restrict production exposure.</action>
  <verify>Start server in production mode (NODE_ENV=production), curl a non-existent route — response must NOT contain availableRoutes array. Curl an error-triggering endpoint — response must NOT contain stack traces.</verify>
  <done>404 handler returns minimal info in production, error handler verified for production safety, no server technology headers leaked</done>
</task>

<task type="auto">
  <name>Task 2: OWASP Top 10 systematic code audit</name>
  <files>backend-express/server.js (if changes needed), any files with identified issues</files>
  <action>Audit each OWASP 2021 category against the codebase. For each, verify existing controls and fix any gaps:

  **A01 - Broken Access Control:**
  - Verify: All /v1/ routes require JWT (requireAuth middleware applied). Check no route accidentally bypasses auth.
  - Verify: RLS policies enforce user_id isolation in Supabase queries. Check services use req.user.id consistently.
  - Verify: No IDOR — case/document operations filter by authenticated user_id.
  - Fix any gaps found.

  **A02 - Cryptographic Failures:**
  - Verify: AES-256-GCM encryption in documentEncryptionService.js uses proper IV/nonce generation (crypto.randomBytes).
  - Verify: HKDF key derivation uses unique salt per user.
  - Verify: No secrets in code or logs (logger.js has sensitive field filter).
  - Verify: HTTPS enforced via HSTS header (Helmet default).

  **A03 - Injection:**
  - Verify: All DB queries use Supabase query builder (parameterized). No raw SQL concatenation.
  - Verify: No eval(), Function(), or template literal execution of user input.
  - Verify: Claude AI prompts don't allow prompt injection via user document text (check prompt construction).
  - Check for command injection in any child_process usage.

  **A04 - Insecure Design:**
  - Verify: Rate limiting on all API routes.
  - Verify: File upload size limits enforced.
  - Verify: No mass assignment — Joi stripUnknown prevents extra fields.

  **A05 - Security Misconfiguration:**
  - Verify: Helmet.js defaults are appropriate.
  - Verify: Debug/dev endpoints not exposed in production.
  - Verify: Default credentials not used.
  - Check /api-docs (Swagger) — should this be restricted in production?

  **A06 - Vulnerable Components:** Handled in 29-01 (npm audit).

  **A07 - Auth Failures:**
  - Verify: JWT validation checks expiry.
  - Verify: Plaid webhook uses signature verification (not just shared secret).
  - Verify: No hardcoded tokens or API keys.

  **A08 - Software & Data Integrity:**
  - Verify: No unsafe deserialization of user input.
  - Verify: Webhook signatures validated before processing.

  **A09 - Logging & Monitoring Failures:**
  - Verify: Auth failures logged.
  - Verify: Sensitive data NOT logged (check logger filter list).
  - Verify: Sentry captures unhandled errors.

  **A10 - SSRF:**
  - Verify: No user-controlled URLs passed to fetch/http calls.
  - Check if Claude API or Plaid API calls could be manipulated.

  For each category: document finding (pass/fail/N-A), fix any issues found. Track findings for the SECURITY-AUDIT.md in 29-03.

  IMPORTANT: Only fix actual vulnerabilities. Do not add speculative defenses for scenarios that can't happen given the architecture (e.g., don't add CSRF tokens to a stateless JWT API).</action>
  <verify>npm test --prefix backend-express passes all tests after any fixes. No new security warnings introduced.</verify>
  <done>All OWASP Top 10 categories audited, findings documented in code comments where fixes applied, all tests pass</done>
