  <name>Task 1: Backend Sentry integration</name>
  <files>backend-express/utils/sentry.js, backend-express/server.js, backend-express/utils/envValidator.js, backend-express/.env.example</files>
  <action>
1. Install Sentry: `cd backend-express && npm install @sentry/node`

2. Create `backend-express/utils/sentry.js`:
   - Import `@sentry/node`
   - Export `initSentry(app)` function that:
     - Checks `SENTRY_DSN` env var — if not set, log info message "Sentry not configured (SENTRY_DSN not set)" and return (graceful no-op)
     - Calls `Sentry.init({ dsn, environment: NODE_ENV, release: '2.0.0', tracesSampleRate: 0.1 (10% of requests), integrations: [Sentry.expressIntegration()] })`
     - Returns Sentry instance for testing
   - Export `sentryErrorHandler()` that returns `Sentry.expressErrorHandler()` if initialized, or a no-op middleware if Sentry not configured
   - Export `addSentryContext(req)` helper that calls `Sentry.setContext('request', { requestId: req.requestId, userId: req.user?.id, method: req.method, path: req.path })` — safe no-op if Sentry not initialized

3. Integrate in `server.js`:
   - Import `initSentry`, `sentryErrorHandler` from utils/sentry
   - Call `initSentry(app)` EARLY — after Express app creation but before routes
   - Add `sentryErrorHandler()` AFTER all route handlers but BEFORE the existing error handler
   - In the existing error handler, call `addSentryContext(req)` before the error response
   - Do NOT remove or change the existing error handler behavior — Sentry is additive

4. Add `SENTRY_DSN` to envValidator.js:
   - Classify as `production-only` tier (not required for dev)
   - Joi: `Joi.string().uri().optional()`

5. Add to `backend-express/.env.example`:
   ```
   # ============================================
   # ERROR TRACKING (production-only)
   # ============================================
   # Sentry DSN for error tracking and performance monitoring
   # Get from: https://sentry.io → Project Settings → Client Keys
   # SENTRY_DSN=https://examplePublicKey@o0.ingest.sentry.io/0
   ```

Do NOT add Sentry as a Winston transport — keep them separate channels (Winston for structured logs, Sentry for error tracking).
Do NOT enable Sentry in test environment — check NODE_ENV !== 'test'.
Do NOT capture PII in Sentry context — only requestId, userId (UUID), method, path.

<task type="auto">
  <name>Task 2: Frontend Sentry integration and env documentation</name>

  <action>
1. Install Sentry: `cd frontend && npm install @sentry/nextjs --legacy-peer-deps`

2. Create `frontend/src/lib/sentry.ts`:
   - Import `@sentry/nextjs`
   - Export `initSentry()` that:
     - Checks `NEXT_PUBLIC_SENTRY_DSN` — if not set, return (no-op)
     - Calls `Sentry.init({ dsn, environment: process.env.NODE_ENV, tracesSampleRate: 0.1, replaysSessionSampleRate: 0, replaysOnErrorSampleRate: 0 })`
     - Disable replay to minimize bundle size
   - Call `initSentry()` at module level so it runs on import

3. Create `frontend/src/app/global-error.tsx`:
   - Next.js App Router error boundary for root layout errors
   - Import Sentry, capture exception
   - Render minimal error UI: "Something went wrong" with retry button
   - Use `'use client'` directive
   - Structure:
     ```tsx
     'use client';
     import * as Sentry from '@sentry/nextjs';
     import { useEffect } from 'react';
     
     export default function GlobalError({ error, reset }: { error: Error; reset: () => void }) {
       useEffect(() => { Sentry.captureException(error); }, [error]);
       return (
         <html><body>
           <h2>Something went wrong</h2>
           <button onClick={() => reset()}>Try again</button>
         </body></html>
       );
     }
     ```

4. Import sentry.ts in `frontend/src/app/layout.tsx`:
   - Add `import '@/lib/sentry'` at the top (side-effect import for initialization)
   - Do NOT change any existing layout structure

5. Add `NEXT_PUBLIC_SENTRY_DSN` to frontend env:
   - In `frontend/.env.example`: add commented-out `# NEXT_PUBLIC_SENTRY_DSN=` with description
   - In `frontend/src/lib/env.ts`: add as optional variable (not required, no validation error if missing)

6. Update `ENV-GUIDE.md`:

   - SENTRY_DSN (backend, production-only) and NEXT_PUBLIC_SENTRY_DSN (frontend, optional build-time)
   - Note: both are optional — app runs fine without them

Do NOT use @sentry/nextjs wizard (it creates too many files) — manual integration is cleaner.
