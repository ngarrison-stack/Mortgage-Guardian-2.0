Enhance the existing health route file to provide three endpoints:

1. **GET /health** (existing — keep backward-compatible):
   - Keep current response shape (status, timestamp, uptime, environment, version, services)
   - Add `checks` object with results from deep checks (see readiness below)
   - If any deep check fails, still return 200 but set `status: 'degraded'` (not 'unhealthy' — the app itself is running)

2. **GET /health/live** (new — liveness probe):
   - Minimal: returns `{ status: 'ok' }` with 200
   - No external checks — only confirms the process is responding
   - Used by Docker HEALTHCHECK and container orchestrators for restart decisions

3. **GET /health/ready** (new — readiness probe):
   - Deep checks: attempt lightweight connectivity to each configured service:
     - **Supabase**: Call `supabase.from('users').select('count').limit(1)` wrapped in try/catch with 3-second timeout. Report connected/disconnected.
     - **Redis**: If ioredis client available, call `redis.ping()` with 2-second timeout. Report connected/disconnected/not-configured.
   - Return 200 if ALL configured required services pass, 503 if any required service fails
   - Response: `{ status: 'ready'|'not_ready', checks: { supabase: { status, latencyMs }, redis: { status, latencyMs } } }`
   - Do NOT import Supabase/Redis directly — use optional require or pass via app.locals so the route stays testable

Update the Docker healthcheck comment in health.js noting that /health/live is preferred for container probes.

Do NOT change the existing / root endpoint.
Do NOT add authentication to any health endpoint.
Do NOT make Redis a required check — it's optional infrastructure.

1. Create `backend-express/middleware/metrics.js`:
   - Export a metrics collector object (singleton) with:
     - `totalRequests`: counter
     - `totalErrors`: counter (status >= 500)
     - `statusCodes`: Map of status code → count
     - `responseTimeHistogram`: Array of recent response times (keep last 1000, ring buffer)
     - `startTime`: Date.now() at module load (for uptime calculation)
   - Export middleware function that:
     - Records start time via `process.hrtime.bigint()` (NOT Date.now — insufficient precision)
     - Hooks into `res.on('finish', ...)` to capture:
       - Response time in ms (round to 2 decimal places)
       - Status code increment
       - Error count if >= 500
     - Do NOT log every request here (Morgan already does that)
   - Export `getMetrics()` function returning snapshot:
     - `uptime`: seconds since start
     - `requests`: { total, errors, errorRate (percentage) }
     - `statusCodes`: Object from Map entries
     - `responseTime`: { avg, p50, p95, p99 } computed from histogram
     - Percentile calculation: sort array, index = Math.ceil(percentile/100 * length) - 1
   - Export `resetMetrics()` for testing

2. Add **GET /metrics** endpoint to `health.js`:
   - Import getMetrics from metrics middleware
   - Return JSON snapshot from getMetrics()
   - No authentication (operational endpoint like /health)
   - Place alongside health routes (same router)

3. Mount metrics middleware in `server.js`:
   - Import and use BEFORE route handlers (after morgan, after requestId)
   - `app.use(metricsMiddleware)` — applies to ALL routes

Do NOT use Prometheus client library — keep it simple JSON for now. External Prometheus scraping can be added later if needed.
Do NOT track per-route metrics — keep it aggregate for v1.
Do NOT persist metrics across restarts — in-memory only is fine for container deployments.
