<task type="auto">
  <name>Task 1: Memory profiling under sustained load</name>
  <files>backend-express/loadtest/suites/memory.js</files>
  <action>
Create `backend-express/loadtest/suites/memory.js` that profiles memory during load:

1. **Before load:** GET /metrics to capture initial memory baseline (process.memoryUsage is available via metrics endpoint — check if exposed, if not, add a /health/debug endpoint that returns process.memoryUsage() only in non-production NODE_ENV)

2. **During load:** Run 50 connections for 30 seconds against /health/live while polling /metrics or /health/debug every 2 seconds to track memory growth

3. **After load:** Wait 5 seconds for GC, poll memory again

4. **Report:**
   - Initial RSS, heap used, heap total
   - Peak RSS during load
   - Post-load RSS (check for memory leak indicator: if post > 1.5x initial, warn)
   - Pass/fail: peak RSS < 100MB target

If the /metrics endpoint doesn't expose memory info, add `process.memoryUsage()` data to the existing /metrics response (rss, heapUsed, heapTotal in bytes). This is a small, safe addition to existing infrastructure from Phase 27.

Register as `--suite memory` in runner.js.
  </action>
  <verify>cd backend-express && npm run dev & sleep 2 && node loadtest/runner.js --suite memory; kill %1</verify>
  <done>Memory suite tracks RSS through load cycle, reports peak vs target, warns on potential leaks</done>
</task>
