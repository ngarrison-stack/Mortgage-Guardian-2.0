<task type="auto">
  <name>Task 1: API route load test suites</name>
  <files>backend-express/loadtest/suites/api.js</files>
  <action>
Create `backend-express/loadtest/suites/api.js` that load tests the main API routes.

Since these routes require JWT auth and hit external services (Supabase, Claude AI, Plaid), the load test operates in two modes:

**Mode 1 — Unauthenticated baseline (default):**
Test endpoints that respond with 401/403 to measure auth middleware overhead:
- POST /v1/documents/process → expect 401
- GET /v1/plaid/accounts/test → expect 401
- POST /v1/ai/claude/analyze → expect 401
This measures Express routing + middleware stack latency without external service calls.

**Mode 2 — Authenticated (if AUTH_TOKEN env var set):**
If AUTH_TOKEN is set, include it as Bearer token in headers and test full request path. Print warning that results depend on external service availability.

For each endpoint group, test with:
- 10 connections, 15 seconds duration
- Record: latency p50/p95/p99, req/sec, error rate
- Print formatted results table

Register this suite in runner.js as `--suite api`.
  </action>
  <verify>cd backend-express && npm run dev & sleep 2 && node loadtest/runner.js --suite api; kill %1</verify>
  <done>API suite runs, measures routing overhead for all main endpoints, prints latency table</done>
</task>

<task type="auto">
  <name>Task 2: Stress test suite with concurrent connections</name>
  <files>backend-express/loadtest/suites/stress.js</files>
  <action>
Create `backend-express/loadtest/suites/stress.js` that tests server behavior under increasing load:

**Ramp-up test on /health/live** (lightest endpoint to isolate server capacity):
- Stage 1: 10 connections, 10 seconds → baseline
- Stage 2: 50 connections, 10 seconds → moderate load
- Stage 3: 100 connections, 10 seconds → heavy load
- Stage 4: 200 connections, 10 seconds → stress test

After each stage, record:
- Req/sec, latency p99, error count, timeouts
- Print comparison table showing degradation across stages

**Rate limit verification:**
Test that rate limiting activates under sustained single-IP load:
- 1 connection, rapid fire for 5 seconds against any authenticated route
- Report whether rate limit (429) responses appear and at what threshold

**Pass/fail criteria:**
- p99 < 500ms at 50 connections
- p99 < 1000ms at 100 connections
- Error rate < 5% at 200 connections (excluding expected 401/429)
- Rate limiter triggers within configured window

Register as `--suite stress` in runner.js.
  </action>
  <verify>cd backend-express && npm run dev & sleep 2 && node loadtest/runner.js --suite stress; kill %1</verify>
  <done>Stress suite runs 4-stage ramp-up, prints comparison table, validates pass/fail criteria</done>
</task>
