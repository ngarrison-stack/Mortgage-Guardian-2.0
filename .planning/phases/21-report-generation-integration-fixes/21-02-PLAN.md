The `disputeLetterSchema` in `consolidatedReportSchema.js` has two structural mismatches with what `disputeLetterService.generateDisputeLetter()` actually returns:

**Mismatch 1 — content field:**
Schema says: `content: Joi.string().required()`
Service returns:
```js
content: {
  subject: string,
  salutation: string,
  body: string,
  demands: string[],
  legalCitations: string[],
  responseDeadline: string,
  closingStatement: string
}
```

**Mismatch 2 — recipientInfo field:**
Schema says: `recipientInfo: Joi.object({ name, address, department })`
Service returns: `recipientInfo: { servicerName, servicerAddress }`

Fix the `disputeLetterSchema` to match the actual service output:

```js
const disputeLetterContentSchema = Joi.object({
  subject: Joi.string().allow('').required(),
  salutation: Joi.string().allow('').required(),
  body: Joi.string().allow('').required(),
  demands: Joi.array().items(Joi.string()).default([]),
  legalCitations: Joi.array().items(Joi.string()).default([]),
  responseDeadline: Joi.string().allow('').required(),
  closingStatement: Joi.string().allow('').required()
}).required();

const disputeLetterSchema = Joi.object({
  letterType: Joi.string().required(),
  generatedAt: Joi.string().isoDate().required(),
  content: disputeLetterContentSchema,
  recipientInfo: Joi.object({
    servicerName: Joi.string().required(),
    servicerAddress: Joi.string().required()
  }).required()
}).allow(null).default(null);
```

Do NOT change the service output — fix the schema to match reality. The schema is the contract; it should describe what the service actually produces.

The dispute letter service is called in two contexts:
1. **During initial generation** (Step 5 in consolidatedReportService) — receives raw aggregated data with `documentAnalyses` (plural), `complianceReport`, `forensicReport`
2. **From POST /letter route** — receives stored consolidated report with `documentAnalysis` (singular), `complianceFindings`, `forensicFindings`

The `_extractViolations(report)` method reads:
- `report.complianceReport.violations` — works for raw, fails for stored (should also check `report.complianceFindings.federalViolations`)
- `report.complianceReport.stateViolations` — works for raw, fails for stored (should also check `report.complianceFindings.stateViolations`)

The `_extractFindings(report)` method reads:
- `report.documentAnalyses` (plural) — works for raw, fails for stored (should also check `report.documentAnalysis`)
- `report.forensicReport.discrepancies` — works for raw, fails for stored (should also check `report.forensicFindings.discrepancies`)

Fix both methods to handle BOTH data shapes by checking the consolidated report field names first (preferred — more structured), then falling back to raw aggregated field names:

In `_extractViolations(report)`:
```js
// Support both consolidated report and raw aggregated data
const complianceData = report.complianceFindings || report.complianceReport || {};
const federalViolations = complianceData.federalViolations || complianceData.violations || [];
const stateViolations = complianceData.stateViolations || [];
```

In `_extractFindings(report)`:
```js
// Support both consolidated report and raw aggregated data
const documentAnalyses = report.documentAnalysis || report.documentAnalyses || [];
const forensicData = report.forensicFindings || report.forensicReport || {};
const discrepancies = forensicData.discrepancies || [];
```

This makes the letter service work correctly regardless of which data shape it receives.
