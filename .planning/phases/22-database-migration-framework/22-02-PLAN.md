<task type="auto">
  <name>Task 1: Create baseline migration from schema dump</name>
  <files>supabase/migrations/00000000000000_baseline.sql</files>
  <action>
    Create the baseline migration file using the Supabase CLI convention.

    Use `supabase migration new baseline` to generate a properly timestamped migration file in `supabase/migrations/`. Then populate it with the schema from the dump.

    **Organization of the baseline migration:**
    1. Extensions (if any from dump)
    2. Functions (update_updated_at_column, etc.)
    3. Tables in dependency order:
       - Independent tables first (plaid_items, notifications)
       - Dependent tables next (plaid_accounts → plaid_items FK, plaid_transactions → plaid_items + plaid_accounts FKs)
       - documents table
       - case_files table
       - document_classifications table (references documents)
       - pipeline_state table (references case_files)
    4. Indexes
    5. RLS enable statements
    6. RLS policies
    7. Triggers
    8. Storage bucket policies (if captured in dump)

    **Important:**
    - Use `CREATE TABLE IF NOT EXISTS` for idempotency
    - Use `DO $$ ... END $$` guards for policies (matching existing pattern in 004/005)
    - Preserve exact column types, defaults, and constraints from the dump
    - Do NOT modify any table structure — capture as-is per CONTEXT.md boundaries
    - Include comments marking sections for readability
    - If the dump includes Supabase system tables (auth.*, storage.* internals), exclude them — only include application schema objects
  </action>
  <verify>
    - Migration file exists in `supabase/migrations/` with proper timestamp prefix
    - File contains all 8 application tables
    - Dependency order is correct (no FK references to tables not yet created)
    - All RLS policies from dump are included
    - All indexes from dump are included
    - All trigger functions and triggers are included
    - No Supabase internal/system objects included
  </verify>

<task type="auto">
  <name>Task 2: Create rollback migration and archive old migrations</name>
  <files>supabase/migrations/00000000000000_baseline_down.sql, backend-express/migrations/_archived/</files>
  <action>
    **Part A — Down migration:**
    Create a rollback SQL file at `supabase/migrations/` (or a `down/` subdirectory — follow Supabase CLI convention for down migrations if one exists, otherwise create `supabase/rollback/baseline_down.sql`).

    The down migration must reverse the baseline in reverse dependency order:
    1. Drop triggers
    2. Drop RLS policies
    3. Disable RLS on tables
    4. Drop indexes
    5. Drop tables in reverse dependency order (pipeline_state, document_classifications, case_files, documents, plaid_transactions, plaid_accounts, notifications, plaid_items)
    6. Drop functions (update_updated_at_column)

    Use `DROP TABLE IF EXISTS ... CASCADE` for safety.
    Use `DROP POLICY IF EXISTS` for policies.
    Use `DROP FUNCTION IF EXISTS` for functions.

    **Part B — Archive old migrations:**
    Move the 5 old manual migration files from `backend-express/migrations/` to `backend-express/migrations/_archived/`:

    Add a README.md in `_archived/` explaining these are the original manual migrations, now superseded by the Supabase CLI migration framework in `supabase/migrations/`.

    Do NOT delete the old files — archive them for historical reference.
  </action>
