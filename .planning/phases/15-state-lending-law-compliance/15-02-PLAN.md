<feature>
  <name>Jurisdiction Detection Service</name>
  <files>backend-express/services/jurisdictionService.js, backend-express/__tests__/jurisdictionService.test.js</files>
  <behavior>
    detectJurisdiction(caseData, options) → { propertyState, servicerState, applicableStates, determinationMethod, confidence }

    Cases:
    - caseData.propertyState = 'CA' → applicableStates includes 'CA', method = 'property_location', confidence = 'high'
    - caseData.servicerState = 'NY' and no propertyState → applicableStates = ['NY'], method = 'servicer_location', confidence = 'medium'
    - caseData.propertyState = 'TX', caseData.servicerState = 'NY' → applicableStates = ['TX', 'NY'] (both may apply), method = 'property_location', confidence = 'high'
    - Unsupported state (e.g., 'WY') → applicableStates = [] (state not in taxonomy), confidence = 'low'
    - No state info at all → applicableStates = [], method = 'default', confidence = 'none'
    - options.manualState = 'FL' → overrides detection, applicableStates = ['FL'], method = 'manual', confidence = 'high'
    - Invalid state code (e.g., 'XX', '123', null) → handled gracefully, excluded from applicableStates
    - Case metadata extraction: scan documents array for state references in addresses
    - Deduplication: same state from multiple sources only appears once in applicableStates
    - isStateSupported check: only include states that have statute data in stateStatuteTaxonomy
  </behavior>
  <implementation>Singleton class with detectJurisdiction() as main method. Uses stateStatuteTaxonomy.isStateSupported() to filter results. Priority order: manual override > property state > case metadata > servicer state. Returns structured jurisdiction object matching the jurisdictionSchema from 15-01.</implementation>
</feature>
