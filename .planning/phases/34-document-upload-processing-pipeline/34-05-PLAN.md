   - Add @State errorMessage: String? and @State showingError: Bool
   - Add .alert("Error", isPresented: $showingError) that shows errorMessage with "OK" and optional "Retry" button
   - Use the userFriendlyError() mapping pattern from Phase 33-03 to translate APIError cases to readable messages

2. Apply this error handling to all API operations in DocumentsView:
   - fetchDocumentsFromBackend() failures → "Couldn't load documents. Pull to refresh to try again."
   - analyzeDocument() failures → "Couldn't start analysis. Tap Analyze to try again."
   - deleteDocument() failures → "Couldn't delete document. Please try again."
   - getDocumentAnalysis() failures → "Couldn't load analysis results. Tap to retry."

3. In DocumentCaptureViewModel, ensure upload failures show clear feedback:

   - Auth errors → "Session expired. Please sign in again." (the 401 callback should handle this, but add a user-facing message too)
   - Server errors → "Upload failed. Please try again."

4. Add a "Retry" option to DocumentsView for documents stuck in a failed state — if a document was uploaded but pipeline failed, show a retry button that calls processDocument() again.

Do NOT add offline queue or background upload — that's beyond this phase scope. Just ensure errors are visible and recoverable via manual retry.
